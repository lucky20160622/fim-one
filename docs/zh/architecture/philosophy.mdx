---## 为什么动态规划是困难的中间地带

AI 代理的生态分成两个阵营，两者都选择了简单的路。传统工作流引擎——Dify、n8n、Coze——选择了静态编排：具有固定执行路径的可视化拖放流程图。这不是无知；企业客户要求确定性（相同输入、稳定输出），而静态图能提供这一点。另一个极端是完全自主的代理（AutoGPT 及其衍生品）承诺端到端的自主性，但被证明不切实际：任务分解不可靠、令牌成本失控，以及没人能预测或调试的行为。

最佳点很窄但真实存在。简单任务不需要规划器。足够复杂需要数十个相互依赖步骤的任务会压倒当前的 LLM。但在两者之间存在一类丰富的问题——具有清晰并行子任务的任务，这些任务很难硬编码，但 LLM 可以轻松分解。动态 DAG 规划正好针对这个区域：模型在运行时提出执行图，框架验证结构并以最大并发运行它。没有拖放，没有盲目自主。## 对改进模型的押注

每隔几个月，基础就会发生变化——GPT-4、函数调用、Claude 3、MCP 协议。在不断变化的基础上构建严格的抽象是有风险的；LangChain 的过度抽象是这个领域中每个人都已经吸取的教训。FIM One 采取相反的方法：**最小抽象，最大可扩展性**。框架拥有编排、并发性和可观测性。智能来自模型，而模型在不断改进。

当今，LLM 任务分解精度对于非平凡目标来说约为 70-80%。当达到 90% 以上时，动态规划的"最佳点"会大幅扩展——昨天过于复杂的问题明天就会变得易于处理。FIM One 的 DAG 框架旨在捕捉这种扩展的价值，而无需重写底层架构。## ReAct 和 DAG 规划会过时吗？

ReAct 不会消失 -- 它会沉入模型中。考虑这个类比：你不需要手写 TCP 握手，但 TCP 并没有消失；它被吸收到操作系统中。当模型足够强大时，思考-行动-观察循环变成模型内部的隐式行为，而不是显式框架代码。这已经在发生：Claude Code 本质上是一个 ReAct 代理，其中循环由模型本身驱动，而不是由外部框架驱动。

DAG 规划的持久价值不是"帮助不够聪明的模型分解任务" -- 而是**并发调度**。即使拥有无限能力的模型，物理学也会施加延迟：10 个 LLM 调用的串行链比 3 个并行波浪慢 10 倍。DAG 是一个工程问题（如何快速可靠地运行事物），而不是智能问题（如何决定运行什么）。重试逻辑、成本控制、超时管理、可观测性 -- 当模型变得更聪明时，这些都不会消失。

最终状态：**模型拥有"什么"（规划智能内化到模型中），框架拥有"如何"（并发、重试、监控、成本治理）**。框架的持久价值不是智能 -- 而是治理。## 为什么不镜像 Dify 的工作流编辑器

一个自然的问题：如果 DAG 涵盖了灵活的情况，我们是否也应该为确定性情况构建一个静态工作流编辑器？

不应该。原因如下：

1. **工作流已经存在于其他地方。** 政府和企业客户的固定流程存在于他们的 OA、ERP 和遗留系统中。他们不想在另一个平台上重建这些流程——他们想要能够连接到这些流程已经运行的系统的 AI。Connector Platform (v0.6) 直接解决了这个问题。

2. **现有功能组合成固定管道。** Scheduled Jobs (v1.0) 使用固定提示触发 DAG agent；DAG 动态规划步骤；Connectors (v0.6) 连接到目标系统。这个组合等同于一个静态管道——但更灵活，因为 LLM 可以根据它遇到的数据调整其计划。不需要单独的管道 DSL。

3. **投资不匹配。** 一个生产级质量的可视化工作流编辑器（画布、节点类型、变量传递、调试/重放）需要数月的专项工作。结果将是 Dify 的 120K 星社区已经维护的低保真副本。这些工作最好花在 Connector 架构上——这是竞争对手没有提供的功能。

战略赌注：**不要在工作流可视化上竞争；要在成为系统与 AI 相遇的枢纽上竞争**。让 Dify 拥有"以可视化方式构建 AI 工作流"。FIM One 拥有"系统与 AI 相遇的枢纽"。## FIM One 的定位

FIM One 不是"AGI 任务调度器"，也不是静态工作流引擎。它处于中间地带：具有约束的规划能力、可观测性的并发。

- 与 **Dify** 相比：更灵活 -- 运行时 DAG 生成 vs. 设计时流程图。您无需提前预测每个执行路径。不在可视化工作流编辑上竞争；在遗留系统集成上竞争。
- 与 **AutoGPT** 相比：更可控 -- 有界迭代、重新规划限制，以及路线图上的人工干预。在护栏内的自主性。

策略很直接：现在构建编排框架，让不断改进的模型随着时间推移为其填充能力。## FIM One 的位置：规划 × 执行频谱

**为什么 Dify/n8n 是静态规划 + 静态执行**：DAG 由人类在设计时在可视化画布上绘制。每个节点执行固定操作（带有固定提示的 LLM 调用、HTTP 请求、代码块）。没有运行时重新规划——如果一个步骤失败，工作流失败或遵循预先连接的错误分支。这在结构上与 BPM 相同，只是图中有 AI 节点。

**FIM One 的位置：动态规划 + 动态执行**

- **DAG 拓扑在运行时由 LLM 生成**（动态规划）——没有人设计图
- **每个 DAG 节点运行完整的 ReAct 循环**（动态执行）——节点推理、使用工具并自适应
- **重新规划机制**（执行 → 分析 → 如果不满意则重新规划）
- 但有界限：最多 3 个重新规划轮次、令牌预算、人工确认门

这将 FIM One 放在与 AutoGPT 相同的象限中，但具有防止失控行为的工程约束。比 BPM/Dify 更灵活，比 AutoGPT 更受控。## 概念术语表

对于不熟悉本文档中使用的术语的读者：

| 术语 | 单行解释 | 与 FIM One 的关系 |
|------|---------------------|----------------------|
| **BPM**（业务流程管理） | 流程在设计时完全固定，严格执行。Camunda、Activiti。 | FIM One **不是** BPM。没有固定的流程引擎。 |
| **FSM**（有限状态机） | 系统在任何时间都恰好处于一个状态；事件触发转换。支持循环（拒绝 → 重新提交）。 | 目标系统（ERP、合同系统）内部使用 FSM。FIM One **不需要**自己的 FSM -- 它调用目标系统的 API。 |
| **ACM**（自适应案例管理） | 骨架静态，分支动态。主流程预定义，每个节点在运行时自适应。 | FIM One 的 DAG + ReAct 自然落在这个象限。 |
| **HTN**（分层任务网络） | 递归任务分解：高级 → 子任务 → 原子操作。 | DAG 重新规划涵盖大多数场景；暂不需要完整的 HTN。 |
| **iPaaS**（集成平台即服务） | 云集成平台连接多个 SaaS/本地系统。MuleSoft、Zapier。 | FIM One 的 Hub Mode 类似于 **AI 原生 iPaaS** -- 自然语言驱动跨系统集成。 |## 架构边界：FIM One 不复制工作流逻辑

复杂的业务流程（审批链、转移、拒绝、升级、共签、副签）是**目标系统的责任**。这些系统花费多年构建这些逻辑 -- ERP、OA、合同管理系统都有成熟的状态机。

从 Connector 的角度来看：

| 操作 | Connector 的作用 |
|-----------|------------------------|
| Transfer | 调用一个 API，传递目标人员 |
| Reject | 调用一个 API，传递拒绝原因 |
| Escalate | 调用一个 API，传递升级人员列表 |
| Co-sign | 调用一个 API，传递共签人员列表 |

每个复杂的工作流操作都简化为一个带参数的 HTTP 请求。FIM One 调用 API；目标系统管理状态机。

这是一个**刻意的架构边界**，不是能力差距。复制目标系统中已存在的工作流逻辑会：
1. 增加维护负担（两个状态机需要保持同步）
2. 引入故障模式（如果它们不一致怎么办？）
3. 提供零附加价值（目标系统已经正确处理了这一点）

Connector 模式设计简洁：**一个操作 = 一个 API 调用**。
