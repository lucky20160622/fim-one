所有配置均通过 `.env` 进行。复制 `example.env` 并填入相应的值：

```bash
cp example.env .env
```

<div id="configuration-levels">
  ## 配置级别
</div>

每个集成都有一个配置级别，用于表明其重要程度：

| 级别 | 含义 | 未配置时的行为 |
|-------|---------|------------------------------|
| **必填** | 核心系统依赖 | 系统会报错——聊天和主要功能将无法使用 |
| **推荐** | 重要功能支持项 | 优雅降级——该功能会明确显示为不可用，但系统仍可运行 |
| **可选** | 增强能力 | 无感降级——系统可正常运行，只是该能力不存在 |

> **注意**：管理员配置的模型（Admin → Models 页面）可替代 LLM 环境变量。健康检查会同时考虑这两种来源。

<div id="frontend-local-dev-only">
  ## 前端（仅限本地开发）
</div>

前端有一个单独的 env 文件，**仅用于本地开发**：`frontend/.env.local`。

> **Docker 中不会使用此文件。** 在 Docker 容器内，Next.js 会在内部将 `/api/*` 代理到 Python 后端（端口 8000 仅在容器内部可用），因此前端不需要 env 文件。

对于本地开发，默认配置开箱即用——你**不**需要创建 `frontend/.env.local`，除非你的后端运行在非默认端口上。

如果需要覆盖默认配置，请手动创建 `frontend/.env.local`：

```bash
echo 'NEXT_PUBLIC_API_URL=http://localhost:9000' > frontend/.env.local
```

| 变量 | 默认值 | 说明 |
|---|---|---|
| `NEXT_PUBLIC_API_URL` | `http://localhost:8000` *(auto)* | **浏览器**用于直接调用 API 的后端 URL（OAuth 重定向、流式传输）。如果未设置，则会从 `window.location` 自动检测——仅当你的后端在本地运行于非标准端口时，才需要覆盖此值。 |

> **构建时说明**：`NEXT_PUBLIC_*` 变量会在执行 `pnpm build` 时固化到 JS bundle 中。在运行时修改它们（例如通过根目录的 `.env`）不会生效——这就是为什么它们仅用于本地开发，并放在 `frontend/.env.local` 中。

<div id="llm-required">
  ## LLM（必填）
</div>

| 变量 | 必填 | 默认值 | 说明 |
|---|---|---|---|
| `LLM_API_KEY` | **是** | — | LLM 提供商的 API 密钥 |
| `LLM_BASE_URL` | 否 | `https://api.openai.com/v1` | 任何兼容 OpenAI 的 API 的基础 URL |
| `LLM_MODEL` | 否 | `gpt-4o` | 主模型——用于规划、分析和 ReAct 智能体 |
| `FAST_LLM_MODEL` | 否 | *(回退到 `LLM_MODEL`)* | 快速模型——用于 DAG 步骤执行（成本更低、速度更快） |
| `LLM_TEMPERATURE` | 否 | `0.7` | 默认采样温度 |
| `LLM_CONTEXT_SIZE` | 否 | `128000` | 主 LLM 的上下文窗口大小 |
| `LLM_MAX_OUTPUT_TOKENS` | 否 | `64000` | 主 LLM 每次调用的最大输出令牌数 |
| `FAST_LLM_CONTEXT_SIZE` | 否 | *(回退到 `LLM_CONTEXT_SIZE`)* | 快速 LLM 的上下文窗口大小 |
| `FAST_LLM_MAX_OUTPUT_TOKENS` | 否 | *(回退到 `LLM_MAX_OUTPUT_TOKENS`)* | 快速 LLM 每次调用的最大输出令牌数 |
| `LLM_REASONING_EFFORT` | 否 | *(禁用)* | 支持扩展思考的模型（OpenAI o 系列、Gemini 2.5+、Claude）的推理强度。可选值：`low`、`medium`、`high`。OpenAI/Gemini 发送 `reasoning_effort` 参数；Anthropic（自动检测）通过 `extra_body` 发送 `thinking`。模型的思维链将在 UI 的"思考中"步骤中显示。 |
| `LLM_REASONING_BUDGET_TOKENS` | 否 | *(根据 effort 自动计算)* | Anthropic thinking 的显式 token 预算（最小 1024）。OpenAI/Gemini 直接使用 effort 级别。仅在设置了 `LLM_REASONING_EFFORT` 时生效。 |

> **解析顺序**：用户偏好 → 管理员模型（DB）→ ENV 回退。如果在 Admin → Models 中配置了角色为 "General" 的管理员模型，则这些 ENV 变量仅作为回退项。健康检查会同时考虑这两种来源。

### 扩展思考（Reasoning）

设置 `LLM_REASONING_EFFORT` 后，FIM Agent 会启用模型的扩展思考能力，并将内部思维链展示在 UI 的"思考中"步骤中。**目前仅支持以下两条官方路径：**

#### 支持的提供商

| 提供商 | `LLM_BASE_URL` | 发送的 API 参数 | 是否返回推理内容？ |
|---|---|---|---|
| **OpenAI**（o1 / o3 / o4-mini） | `https://api.openai.com/v1` | `reasoning_effort: "high"` | 是 |
| **Anthropic**（Claude 3.7+） | `https://api.anthropic.com/v1/` | `extra_body: {thinking: {type: "enabled", budget_tokens: N}}` | 否（兼容端点限制） |

系统通过检查 `LLM_BASE_URL` 是否包含 `anthropic.com` 来自动检测 Anthropic。

- **Google Gemini** 在其 OpenAI 兼容端点（`generativelanguage.googleapis.com/v1beta/openai/`）上也支持 `reasoning_effort`，理论上可以直接使用——但我们尚未正式测试。

#### 重要注意事项

<Warning>
**Anthropic 的 OpenAI 兼容端点不会返回推理内容。**
如果你将 `LLM_BASE_URL` 设置为 Anthropic 的官方端点（`https://api.anthropic.com/v1/`），`thinking` 参数会正确发送，模型*确实会*进行内部思考，但响应中**不会包含推理文本**——你只能看到最终回答。UI 的"思考中"步骤会出现，但推理内容为空。这是 Anthropic 兼容层的限制，不是 bug。

如需完整的 Claude 推理可见性，请使用原生 Anthropic SDK 或能返回 `reasoning_content` 的第三方代理。
</Warning>

<Warning>
**第三方代理 / 自定义端点不保证兼容。**
如果你的 `LLM_BASE_URL` 指向第三方 API 代理（如 OpenRouter、one-api、自定义网关），`reasoning_effort` 参数会原样发送。代理能否正确将其转换为底层提供商的格式，完全取决于代理的实现。部分代理可能期望不同的格式（如 `reasoning: {effort, max_tokens}`）。如果推理功能在你的代理上无法工作，请查阅代理的文档了解其期望的参数格式——FIM Agent 无法适配任意代理的自定义协议。
</Warning>

#### `LLM_REASONING_BUDGET_TOKENS` 的工作方式

此变量**仅对 Anthropic 路径有效**。设置后，它会覆盖自动计算的预算，作为 `budget_tokens` 发送到 `thinking` 参数中。未设置时，预算根据 `LLM_MAX_OUTPUT_TOKENS` × effort 比例自动计算：

| `LLM_REASONING_EFFORT` | 预算比例 | 示例（max_tokens = 64000） |
|---|---|---|
| `low` | 20% | 12,800 tokens |
| `medium` | 50% | 32,000 tokens |
| `high` | 80% | 51,200 tokens |

最低预算为 1,024 tokens（Anthropic 的硬性下限）。

对于 OpenAI 和 Gemini，提供商根据 `reasoning_effort` 级别内部处理 token 分配——`LLM_REASONING_BUDGET_TOKENS` 无效。
