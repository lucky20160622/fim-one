> **状态**: 已接受（2026年3月）
>
> **背景**: 随着LLM能力的快速演进，我们需要一个框架来决定在哪里投入工程力量，以及在哪里保持不变。## 决策

我们根据每项功能与 LLM 进步的关系对其进行分类，并相应地分配工作量。

| 类别 | 策略 | 投资 | 示例 |
|----------|----------|------------|---------|
| **正交** | 模型变得更聪明不会削弱这些功能 — 纯工程/集成问题 | 全力投资 | Connectors、凭证、OAuth、审计、RBAC、安全、部署 |
| **Tailwind** | 模型改进使这些功能*更好*，而非冗余 — 共生关系 | 投资（收益复合） | AI Connector Builder（更聪明的模型 = 更高质量的连接器输出） |
| **冻结** | 已发布、运行良好 — 但模型正在吸收这些功能 | 仅维护，无新功能 | ReAct 循环、DAG 规划、RAG 管道、Memory、Grounded Generation |
| **考虑** | 提供商在平台级别本地构建 — 冗余风险高 | 无限期延迟 | 多代理编排、语义记忆、记忆生命周期 |

**经验法则**：如果功能解决的是*"如何让模型更聪明"*，它正在被吸收。如果解决的是*"如何安全地将模型连接到真实世界"*，它是正交的。## 分析### 为什么 Connector Platform 完全正交

模型永远不会原生支持：
- 存储和加密 API 凭证 (AES-GCM)
- 管理 OAuth 流程（授权页面 → 回调 → 刷新令牌）
- 连接到客户端的 Kingdee/金蝶 ERP 数据库
- 推送通知到 Lark/飞书 或 WeCom/企微
- 对谁可以使用哪个连接器强制执行 RBAC
- 记录每个工具调用以进行合规审计

这些是工程问题，而不是智能问题。即使一个智能程度高 10 倍的模型，如果没有基础设施支持，也无法做到这些事情。### 为什么 AI Connector Builder 是"顺风车"而不是"被吸收"

Builder Agent 使用模型智能来创建**受管的、持久的 Connector 实体** — 存储在数据库中、可在代理之间重用、具有凭证管理和审计跟踪。模型不断改进的 API 理解能力使 Builder 生成*更好的* Connector，而不是使 Builder 变得不必要。

类比：Cursor 使用 Claude 来编写代码。Claude 变得更聪明使 Cursor *更好*，而不是*冗余*，因为 Cursor 提供了模型无法替代的工程价值（项目管理、文件组织、版本控制）。### 为什么 v0.1–v0.5 功能被"冻结"

| 功能 | 行业发生的情况 |
|---------|--------------------------------|
| ReAct loop | 模型具有原生工具调用能力（OpenAI、Anthropic）。外部推理循环的价值随着模型内化这一能力而降低。 |
| DAG Planning | 模型推理能力快速提升。复杂任务分解曾需要外部规划器，现在成为单次能力。 |
| Memory management | 上下文窗口快速增长（Gemini 2M+、Claude 200K+）。对外部窗口管理、摘要和压缩的需求在缩小。 |
| RAG pipeline | 提供商在其平台中构建检索功能（OpenAI file_search、Google NotebookLM、Gemini Search Grounding）。对于公开知识，传统的分块-嵌入-检索管道正被替代。 |
| Grounded Generation | 模型在原生引用来源方面表现更好。我们构建的 5 阶段接地管道价值递减。 |

**这些功能并不差** — 它们已发布、可正常工作、使产品今天可用。这个决定只是停止对它们的添加，并重新调配工作方向。### 为什么多代理编排被推迟

LLM 提供商正在原生构建编排功能：
- **OpenAI Swarm**：具有切换协议的多代理框架
- **Anthropic Claude Code Teams**：具有任务图的领导者/工作者代理池
- **Google A2A（Agent-to-Agent）**：代理间通信协议

构建竞争性编排层意味着要与具有更深层模型集成的第一方实现竞争。这不是一个可持续的差异化因素。### 为什么语义记忆和记忆生命周期被推迟

- 上下文窗口增长迅速，减少了跨会话记忆检索的需求
- 提供商正在添加原生记忆功能（ChatGPT Memory、Claude Projects）
- 构建可靠记忆系统的工程成本（TTL、重要性评分、语义检索）相对于其填补的缩小差距来说很高## 功能级分类### 正交性 (v0.6+)

1. **不要回到 v0.5 功能。** 可以进行错误修复，但不能添加新功能。
2. **连接器平台是核心投资。** v0.6–v0.8 应该获得大部分工程时间。
3. **企业工程（RBAC、审计、安全、部署）是护城河。** 这些虽然乏味但具有防守性。
4. **每年重新评估。** 如果模型进展停滞或"冻结"的功能仍然存在重大缺陷，应重新考虑。
