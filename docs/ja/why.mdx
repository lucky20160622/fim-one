## すべての企業が抱える問題

50人のスタートアップから Fortune 500 企業まで、すべての企業の背後には、長年にわたって蓄積されたシステムの複雑な絡み合いがあります：

```
ERP (SAP, Kingdee, Yonyou)        — finance, inventory, supply chain
CRM (Salesforce, HubSpot)         — customers, deals, pipelines
OA  (Seeyon, Weaver, Feishu/Lark) — approvals, documents, workflows
HR  (Workday, BambooHR)           — payroll, org charts, leave
Custom databases                   — orders, assets, tickets
Legacy systems                     — "nobody dares touch it"
```

各システムは**情報サイロ**です。データはある場所に存在し、ワークフローは別の場所に、承認はさらに別の場所にあります。人々は何時間もかけてシステム間でコピー&ペーストを繰り返し、スクリーンショットをメールで送信し、自動的に流れるべきデータを手動で調整しています。

**すべてのシステムベンダーが今、独自の AI を提供しています。** Feishu には AI があります。Salesforce には Einstein があります。SAP には Joule があります。しかし、それぞれは**独自の壁の中でのみ**スマートです。Feishu AI は ERP の残高を確認できません。Salesforce Einstein は OA の承認をトリガーできません。これらは設計上、ドメインロックされています。競合他社への接続は彼らのビジネス利益に反するからです。

## FIM Oneの役割

FIM Oneは、**外部の第三者による水平型プラットフォーム**で、すべてのシステムを接続します。

*アバター*のアイウァを思い浮かべてください — パンドーラの下にある神経ネットワークで、すべての木、すべての生き物、すべての記憶を1つの生きたシステムに接続しています。各生物は独立したままですが、アイウァを通じて互いを感知し、情報を共有し、必要に応じて一緒に行動します。生物は変更されません。アイウァは単に接続を可能にするだけです。

FIM Oneはあなたの企業のアイウァです。ERP、CRM、OA、データベースはそのままの状態を保ちます。FIM Oneはそれらの間に知的ネットワークを織り込み、AIを通じてようやく相互に通信できるようにします。

![FIM One — すべての企業システムを接続するアイウァ](images/eywa.jpg)

<Tip>
**クライアントはコードを一切変更しません。** FIM Oneは既存システムにプロアクティブに接続します — データベースを読み取り、APIを呼び出し、通知をプッシュします — あなたが提供する認証情報とネットワークアクセスのみを使用します。
</Tip>

あなたが提供するもの:
- データベース認証情報（読み取り専用アカウントを推奨）
- APIエンドポイントと認証キー
- ネットワークホワイトリストアクセス

FIM Oneが提供するもの:
- **すべての**システムを理解し、オーケストレーションするAI
- すべての操作が監査され、すべての書き込みが人間の確認によってゲートされます
- データはシステムに留まります — FIM Oneはインテリジェンスレイヤーを提供し、データレイクではありません

## 2つのモード、1つのプラットフォーム

| モード | 説明 | 始める |
|------|-----------|------------|
| **Copilot** | 1つのシステム内に組み込まれたAI — ユーザーの既存UIで並行して動作 | 価値を迅速に証明：ERPに「Finance Copilot」を組み込む |
| **Hub** | すべてのシステムを接続する中央AIポータル — クロスシステムオーケストレーション | 価値をスケール：ERP + CRM + OA + Larkを1つの場所で接続 |

**Land and Expand**: 1つのシステムでCopilotを開始します。ユーザーは新しいツールを学ぶことなく、すぐに価値を実感できます。ROIが証明されたら、Hubをセットアップします — すべてのシステムがAIと出会う中央ポータルです。Copilotは実行し続け、Hubはクロスシステムインテリジェンスを追加します。

> Copilotは1つのシステム内で価値を証明します。Hubはすべてのシステム全体で価値を解き放ちます。

## なぜ単に...ではなく

### ...MuleSoftまたは従来のiPaaSを使用するか？

MuleSoftはシステムをシステムに接続します。FIM Oneはシステムを**AI**に接続します。従来のiPaaSには、数ヶ月のカスタムマッピング、専任の統合チーム、および6桁の契約が必要です。FIM Oneはあなたのデータを意味的に理解します。自然言語で何をしたいかを説明すれば、AIがどのシステムをクエリするか、そして結果をどのように組み合わせるかを判断します。

### ...ETLパイプラインを自分で書きますか?

ETLはソースシステムがスキーマをアップグレードするたびに壊れます。FIM Oneはライブで接続します — データコピーなし、同期ジョブなし、パイプラインメンテナンスなし。そして双方向で動作します：あるシステムから読み取り、別のシステムに書き込み、すべての書き込みで人間の確認が必要です。

### ...Dify、n8n、またはワークフロービルダーを使用しますか?

ワークフロービルダーでは、ビジネスロジックを新しいキャンバスで**複製**する必要があります。しかし、ワークフローはすでに存在しています — OA、ERP、承認システムの中に。別のフローチャートエディターは必要ありません。すでに実行されているシステムに接続するAIが必要です。

FIM Oneはワークフロー ロジックを複製しません。コネクターの観点からすると、転送、却下、エスカレーション、副署名などのすべての複雑な操作は、パラメーターを持つ単一のAPI呼び出しに集約されます。ターゲットシステムが状態マシンを管理します。FIM Oneはただ API を呼び出すだけです。

### ...AI エージェント プラットフォーム (Manus、AutoGPT) を使用する?

自律型エージェントは強力ですが、制御不能です。失敗した API の再試行でトークンを消費し、目標から逸脱し、カスタム統合なしにはプロプライエタリ システムに接続できません。FIM One は**制限された自律性**を提供します — ガードレール付きの動的計画、失敗したコネクタ上のサーキット ブレーカー、および書き込み操作に対するヒューマン確認ゲート。

### ...各ベンダーの組み込みAIに任せてはいけないのか？

Feishu AIはFeishu内で動作します。Salesforce Einsteinはsalesforce内で動作します。どちらも境界を越えることはできません。**どのベンダーも真の クロスシステム統合を構築しません** — それはロックイン戦略に反するからです。FIM Oneは、その利益相反がない唯一のプレイヤーです。

## FIM One の接続方法

FIM One は、トランスポートとガバナンスを分離する 3 層アーキテクチャを使用しています：

```
FIM One Platform (AI オーケストレーション、マルチテナント、UI)
    ↓
Connector Governance Layer (監査、認証、確認ゲート、サーキットブレーカー)
    ↓
Connector (API / Database / MCP protocol)
    ↓
Your Systems (ERP, CRM, OA, databases, custom APIs)
```

**ガバナンスレイヤー**がエンタープライズ対応を実現します：

| 機能 | 説明 |
|-----------|-------------|
| **監査ログ** | すべてのツール呼び出しを記録 — 誰が、何を、いつ、結果 |
| **人間による確認** | 書き込み操作は実行前に明示的なユーザー承認が必要 |
| **サーキットブレーカー** | システムがダウンした場合、FIM One はトークンを消費し続ける代わりに即座に再試行を停止 |
| **認証情報の暗号化** | AES-GCM 暗号化（保存時）、ユーザーごとの認証情報分離 |
| **読み取り専用の強制** | デフォルトは読み取り専用。書き込み操作は明示的にゲート化 |

エージェントは、SAP、PostgreSQL データベース、またはカスタム REST API のいずれと通信しているかを知る必要がなく、気にしません。コネクタは標準化されたブリッジです — 新しいシステムを追加するには、コネクタ設定のみが必要で、コード変更は不要です。

## コネクタを構築する3つの方法

Pythonを書く必要なく、新しいシステムを接続できます：

| 方法 | 時間 | 必要なスキル |
|--------|------|---------------|
| **OpenAPI仕様をインポート** | 数分 | YAML/JSON仕様またはURLをアップロード — コネクタが自動生成されます |
| **AIチャットビルダー** | 数分 | APIを自然言語で説明 — AIがコネクタ設定を生成します |
| **MCPエコシステム** | 数分 | 任意のMCPサーバーを直接接続 — オープンソースのMCPコミュニティがそのまま機能します |

エンタープライズデータベースの場合、FIM OneはPostgreSQL、MySQL、Oracle、SQL Server、および中国のレガシーデータベース（DM、KingbaseES、GBase、Highgo）への直接SQLアクセスをサポートしており、スキーマ内省、AI搭載の注釈付け、読み取り専用クエリ実行が可能です。

## なぜ今なのか

FIM Oneを今日可能にした2つの変化があります：

**1. AIモデルがようやくオーケストレーションに十分なほど賢くなった。** GPT-4、Claude、Geminiはネイティブなツール呼び出し、長いコンテキストウィンドウ（200K以上）、および動的なクロスシステムオーケストレーションを信頼できるものにする推論機能を備えています。3年前、これは研究課題でした。今日では、エンジニアリング課題です。

**2. 接続レイヤーはまだ誰の仕事でもない。** モデルはどんどん賢くなっていますが、データベース認証情報をネイティブに暗号化したり、OAuthトークンを管理したり、RBACポリシーを適用したり、ツール呼び出しを監査したりすることはできません。それはインフラストラクチャの仕事です — そしてそれはFIM Oneが行う仕事です。

> 純粋なLLMオーケストレーションから企業システム統合に向かうほど、競争優位性はより堅牢になります。

AIモデルは知能を所有しています。FIM Oneは接続を所有しています。より賢いモデルはFIM Oneを**冗長にするのではなく、より良くします** — より優れた推論は、同じコネクタを通じたより優れたクロスシステムオーケストレーションを意味するからです。
