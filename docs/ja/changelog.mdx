このプロジェクトへのすべての重要な変更はここに記載されています。

形式は[Keep a Changelog](https://keepachangelog.com/)に従っています。バージョンは[ロードマップ](/roadmap)のマイルストーンに対応しています。

## [未リリース]

### Changed

- **ツール出力の一元化された切り詰め** — すべてのツールタイプ（WebFetch、HttpRequest、ShellExec、PythonExec、NodeExec）は、インラインロジックの代わりに共有の `truncation.py` モジュールに切り詰めを委譲するようになりました。2つのエントリポイント：`truncate_tool_output()`（JSON対応、文字レベル）はウェブ/APIツール用、`truncate_bytes()`（バイトレベル）はコード実行ツール用です。デフォルトは `TOOL_OUTPUT_MAX_CHARS`、`TOOL_OUTPUT_MAX_ITEMS`、`TOOL_OUTPUT_MAX_BYTES` 環境変数で設定可能です。
- **ドメイン検出が独立したミドルウェアに分離** — ドメイン分類（法律、医療、金融など）は、自動ルーティング分類にバンドルされるのではなく、各エンドポイント（ReActおよびDAG）内で独立して実行されるスタンドアロンモジュール（`domain.py`）になりました。両エンドポイントは `classify_domain()` を直接呼び出し、自動ルーターが `domain_hint` を伝播する依存性を削除します。
- **ドメインSOP指示の緩和** — 専門分野の指示は、ウェブ検索を強制するのではなく、ガイドするようになりました。ユーザーのクエリとコンテキストに既に十分な情報が含まれている場合、エージェントは不要な検索を強制する代わりに直接回答します。
- **`web_search` は常にReActで固定** — `web_search` ツールは、専門分野が検出されたかどうかに関わらず、常にReActツール選択に含まれるようになり、ツールセレクターによって削除されることがなくなりました。
- **`AUTO_ROUTING` 環境変数を削除** — 自動エンドポイントは常にクエリを分類するようになり、分類を無効にするトグルが削除されました。
- **引用検証者が上流の `domain_hint` を受け入れる** — `should_verify_citations()` は、クエリレベルのLLM分類から `domain_hint` パラメータを受け入れるようになり、ドメインが既知の場合はステップごとのキーワードスキャンをバイパスします。

### 追加

- **モデルごとのネイティブ関数呼び出しトグル** — 新しい `tool_choice_enabled` 設定（ENV グローバル + 管理者によるモデルごと）は、構造化出力抽出のための強制 `tool_choice` が試行されるかどうかを制御します。強制ツール選択を拒否するモデル（例：思考モードモデル）は、レベル 1 をスキップして JSON モードに直接進むことができ、構造化呼び出しごとの約 10 秒のペナルティを排除します。設定 → モデル → 詳細で設定可能です。
- **構造化出力互換性テーブル** — モデルドキュメント内に、ネイティブ関数呼び出し、JSON モード、および計画信頼性評価をサポートするプロバイダーを示すテーブルが追加されました。
- **ファイル添付ダウンロード** — チャットメッセージ内のファイル添付カードがクリック可能になり、認証されたフェッチを介して元のファイルをダウンロードできるようになりました。ダウンロードアイコンインジケーターとホバー状態が発見性のために追加されました。
- **インテリジェント文書処理（ビジョン対応）ロードマップ項目** — モデルのビジョン機能に基づいた適応的な文書処理の計画機能が追加されました（3 つのティア：ビジョンファースト、ハイブリッド、テキストのみ）。
- **DAG 品質オーバーホール（P0）：デフォルトモデルアップグレード** — 明示的な `model_hint` を持たない DAG ステップは、高速モデルではなくレジストリの一般モデル（例：Sonnet）を使用するようになりました。`model_hint="fast"` を持つステップのみが軽量モデルを使用します。
- **DAG 品質オーバーホール（P1）：スキル自動検出** — `DAGPlanner.plan()` は `skill_descriptions` パラメータを受け入れます。利用可能なスキル（名前 + 説明）が計画プロンプトに注入されるため、プランナーはドメインマッチステップに対して `tool_hint="read_skill"` を提案できます。
- **DAG 品質オーバーホール（P2）：引用検証者** — 新しい `citation_verifier.py` モジュールは正規表現を介して法律/医療/金融の引用を抽出し、LLM 判定を介して精度を検証し、修正フィードバックで失敗時に再試行します。エグゼキューター ポストステップフローに統合されています。`DAG_CITATION_VERIFICATION` 環境変数でゲート制御されています（デフォルト：`true`）。
- **DAG 品質オーバーホール（P3）：構造化コンテンツコンテキスト保持** — `_build_step_context()` は依存関係結果内の構造化コンテンツ（法律引用、マークダウンテーブル、コードブロック）を検出し、設定可能な切り詰め乗数（`DAG_STRUCTURED_CONTEXT_MULTIPLIER`、デフォルト `3.0`）を適用して、DAG ステップ間での重要な引用の喪失を防ぎます。
- **DAG 品質オーバーホール（P4）：ルーターのドメイン分類** — `RouteDecision` に `domain_hint`（法律/医療/金融/null）が含まれるようになりました。密結合されたドメイン分析は ReAct モードを優先します。DAG が選択されると、ドメインコンテキストがプランナーに注入され、`model_hint` の選択をガイドします。
- **ReAct でのドメインモデルエスカレーション** — ルーターが専門ドメイン（法律/医療/金融）を検出すると、ReAct エンドポイントは `registry.get_by_role("reasoning")` を介して一般モデルから推論モデルにエスカレートします。Web 検索を書き込み前に実行し、検索を介した引用検証を要求する必須指示が注入されます。`read_skill` ツールはツール選択にピン留めされます。

### 変更内容

- **温度ヒントがツールチップに移動** — モデルフォームダイアログの温度推論ヒントが、スライダーの下の静的段落ではなく、ラベルの横のコンパクトなツールチップアイコン（HelpCircle）として表示されるようになり、視覚的なクラッターが削減されました。
- **RAG グラウンディングパイプライン** — 3段階のパイプライン：検索、引用抽出、信頼度スコアリング。信頼度スコアは、上位3つの証拠ユニットの平均検索スコアです。
- **API キーバナーが「Active」に更新** — 管理者およびユーザー設定の API キーページに、緑色の「API Keys Active」バナー（以前の「Coming Soon」通知に代わる）が表示され、API 認証ドキュメントへのリンクが含まれるようになりました。
- **マルチワーカードキュメントが明確化** — `REDIS_URL` および `WORKERS` 環境変数の説明が、停止、インジェクト、PostgreSQL 要件、およびローカル開発ガイダンスをカバーする警告チェックリスト付きで更新されました。
- **DAG ステップのデフォルトモデル** — `model_hint` のないステップは、汎用モデルに解決されるようになりました（以前はファストモデル）。ファストモデルは `model_hint="fast"` が明示的に設定されている場合にのみ使用されます。
- **ルーター分類出力** — `RouteDecision` は、既存の `mode` フィールドと共に `domain_hint`（legal/medical/financial/null）を出力するようになり、ダウンストリームのドメイン対応モデル選択が可能になりました。

### 修正

- **ロケール ドキュメント内の重複 OpenAPI エンドポイント** — `docs.json` 内のすべての非英語ロケール API タブから冗長な `Endpoints` OpenAPI グループを削除しました。英語セクションは単一のソースを保持し、ロケール セクションはそれを複製しなくなります。
- **構造化出力の劣化チェーン** — 3 レベルのフォールバック (ネイティブ FC → JSON モード → プレーン テキスト抽出) は、モデルが不正な形式の JSON を生成した場合、すべてのレベルを適切に通過するようになりました。以前は Level 2 で `ValueError` で短絡していました。
- **`json_mode_enabled` DB 値が無視される** — Admin (モデル グループ) 経由で構成されたモデルは、`LLM_JSON_MODE_ENABLED` 環境変数に常にフォールバックするのではなく、モデルごとの `json_mode_enabled` 設定を正しく使用するようになりました。
- **DAG プランニング失敗メッセージ** — プランニング失敗は、生のパイプライン エラーの代わりに、モデル切り替えを提案するユーザーフレンドリーな二言語メッセージを表示するようになりました。
- **DAG ステップ カード レイアウト** — `StepProgressCard` を再設計し、タスク テキストがステップ バッジの下に折り返されるようにしました。単一行で切り詰められるのではなく、ステップ ノード内のツール バッジは、`tools_used` が存在しないが `tool_hint` が存在する場合にのみ表示されるようになり、冗長な表示を回避します。
- **MCP サーバー所有者の allow_fallback バイパス** — サーバー所有者は `allow_fallback=False` 制限によってブロックされなくなりました。以前は、`allow_fallback=False` が設定されていた場合、MCP サーバーの作成者でさえスキップされる可能性がありました。現在は、認証情報を持たない所有者以外のユーザーのみが除外されます。
- **モデル プロバイダー API キー検証** — OpenAI 互換でないプロバイダーを作成する場合、インライン検証エラー、必須アスタリスク インジケーター、および適切な `aria-invalid` アクセシビリティ属性を備えた API キーが必要になります。

## [v0.8] - 2026-03-20

### 追加

- **ReAct合成でのエージェント指令** — `ReActAgent`の新しい`agent_directive`パラメータは、エージェントのコア目的（例："中国語を英語に翻訳"）を合成システムプロンプトに注入し、最終回答がエージェントのアイデンティティを尊重し、会話型の返信にデフォルトしないようにします。ReActとDAGチャットエンドポイントの両方に配線されています。

- **マーケットプレイスの再設計 フェーズ1 — ソリューション+コンポーネント2層モデル** — フラットな6タイプのマーケットタブを構造化された2レベルナビゲーションに置き換えました：トップレベルカテゴリトグル（ソリューション：エージェント/スキル/ワークフロー；コンポーネント：コネクタ/MCPサーバー）とタイプサブタブ。グローバルマーケットとユーザーの組織を切り替えるスコープセレクタドロップダウンを追加しました。KBはマーケットスコープから削除されました（組織のみ）。組織とマーケット共有を統一：両方が明示的なサブスクリプションを必要とするようになりました（組織の自動表示はなくなりました）。

- **スマートファイルコンテンツ注入** — 小さなアップロードファイル（`<32K`文字）は自動的に取得されてLLMコンテキストにインライン化されます。大きなファイルはメタデータサマリーとオンデマンド読み取り用のツールヒントを受け取ります。
- **`read_uploaded_file`エージェントツール** — デュアルモードファイル読み取りツール：シーケンシャルアクセス用のページネーション読み取り（オフセット/リミット）とターゲットコンテンツ抽出用のgrepのような検索（正規表現サポート付きクエリ）。
- **`GET /api/files/{file_id}/content`エンドポイント** — アップロードされたファイルから抽出された完全なテキストにアクセスするためのページネーション付きテキストコンテンツAPI。
- **`.content`サイドカーファイル** — 完全に抽出されたテキストはアップロードと並んでサイドカーファイルとして保存され、再抽出なしで永続的なコンテンツアクセスを実現します。
- **ファイルAPIレスポンスの`content_length`フィールド** — ファイルアップロードとリストエンドポイントにテキストコンテンツサイズの可視性のための`content_length`が含まれるようになりました。
- **`fileApi.getContent()`フロントエンドクライアントメソッド** — フロントエンドからページネーション付きファイルコンテンツを取得するための新しいAPIクライアントメソッド。

- **ワークフローエンジンノード再試行** — 指数バックオフを伴うノード単位の再試行。ノードはデータ設定で`retry_count`と`retry_delay_ms`を指定できます。失敗したノードは最大N回、遅延は60秒でキャップされて再試行されます。試行番号と前のエラーを含む`node_retrying` SSEイベントを発行します。再試行待機中のキャンセルイベントを尊重します。
- **ワークフロー変数ストア式評価** — simpleeval経由の安全なPython式評価。`evaluate_expression()`メソッドはストア変数（envを除く）を一般的なビルトイン（len、str、int、float、bool、abs、min、max、sum、round、sorted）で公開します。さらに`has()`、`delete()`、`keys()`ヘルパーメソッド。
- **ワークフロー フェーズ2 ノード：HumanIntervention** — 承認待機ノード（自動承認スタブ）。prompt_message（`{{variable}}`補間付き）、assignee、timeout_hoursを読み取ります。プロンプトプレビューと@assigneeバッジ付きのSky-500テーマフロントエンドノード。プロンプトテキストエリア、assignee入力、タイムアウトフィールド付きの設定パネル。
- **ワークフローテンプレートを12に拡張** — 4つの新しいテンプレート：Human Approval Pipeline（LLM → HumanIntervention → ConditionBranch → デュアルEnd）、Multi-Step Data Processing（CodeExecution → ConditionBranch → Transform → ListOperation）、Knowledge-Enhanced Agent（QuestionUnderstanding → KnowledgeRetrieval → LLM）、Iterative Review Loop（Loop → LLM → VariableAssign）。
- **ワークフロー ペイン コンテキストメニュー** — キャンバスの空いている場所を右クリックして、貼り付け（クリップボードが空の場合は無効）、すべて選択、ビューに合わせる、自動レイアウトアクションにアクセスします。判別された共用体状態がノードとペインコンテキストメニューを分離します。
- **ワークフロー ノード コンテキストメニュー** — 任意のノードを右クリックして、設定、コピー、複製、削除アクションにアクセスします。削除アクションはStart/Endシングルトンノードでは非表示です。メニューは外側のクリックまたはスクロール時に自動的に閉じます。
- **ワークフロー フェーズ2 テンプレート** — 2つの新しい組み込みテンプレート（合計8個）：List Processing Pipeline（Start → ListOperation → Transform → End）とEnhanced Question Answering（Start → QuestionUnderstanding → LLM → End）。
- **ワークフロー フェーズ2 検証** — ListOperation（入力/式がない）、Transform（入力がない/操作が空）、Iterator（リスト変数がない）、Loop（条件がない）、DocumentExtractor、QuestionUnderstanding（入力がない）、ParameterExtractor（入力テキスト/パラメータが空がない）の拡張ブループリント検証警告。修正：ソート操作は式を必要としなくなりました。
- **ワークフロー ノート** — ノード単位の注釈機能。任意のノードの設定パネルで「Note」をクリックして、キャンバス上のノードの下にイタリック体で表示される自由形式のノートを追加します。自動拡張テキストエリア（`data.note`ストレージ）。
- **ワークフロー グリッドスナップ** — ドット背景パターンに一致する16pxスナップグリッドで、ノードの正確な配置を実現します。
- **ワークフロー 接続検証** — 既存のStart/End制限に加えて、自己接続と重複エッジを防止します。
- **ワークフロー ズーム対応ドロップ** — ノードパレットのドラッグアンドドロップは`screenToFlowPosition()`を使用して、任意のズームレベルで正しい配置を実現します。
- **ワークフロー SSE マルチラインデータ** — SSEパーサーはマルチライン`data:`フィールドを処理し、キープアライブコメント行を無視するようになりました。
- **ワークフロー キーボードショートカット** — コピー（Cmd+C）、貼り付け（Cmd+V）、複製（Cmd+D）、すべて選択（Cmd+A）、Escapeで選択解除。コピー/貼り付けは新しいIDを生成し、位置をオフセットします。Start/Endノードはシングルトン制約を維持するためにコピーから除外されます。
- **ワークフロー インライン説明編集** — ツールバーの説明行をクリックして、ワークフロー説明を編集するためのポップオーバーテキストエリアを開きます。
- **ワークフロー ConditionBranch LLMモード** — ナイーブな「最初にマッチ」スタブを置き換える適切なLLMベースの条件評価。条件ラベルから分類プロンプトを構築し、高速LLMを呼び出し、大文字小文字を区別しないファジーマッチングを使用します。
- **リアクティブエッジラベル** — 条件/分類器ブランチのエッジラベルは`useNodesData()`を使用して、設定パネルで条件またはクラスが編集されるときにリアルタイム更新を行います。
- **ワークフロー 統計パネル** — シートベースの統計表示：総実行数、完了/失敗/キャンセル数、ビジュアルプログレスバー付きの成功率、平均期間、相対フォーマット付きの最後の実行時間。ツールバーのBarChart3アイコンからトリガーされます。
- **フェーズ2 ワークフロー ノード：DocumentExtractor & QuestionUnderstanding** — 2つのノードタイプ：
  - **DocumentExtractor**：ドキュメントテキスト処理用のスタブエグゼキューター — テキスト/base64入力タイプと4つの抽出モード（full_text、page_range付きのpages、単語/文字/行/ページ数を含むmetadata、マークダウンテーブル検出を含むtables）をサポートします。URL入力は後の実装のために延期されています。
  - **QuestionUnderstanding**：4つのモード付きLLMベースの質問前処理 — rewrite（明確性）、expand（コンテキスト+サブ質問）、classify（intent/topic/confidence JSON）、decompose（サブ質問配列）。オプションのカスタムシステムプロンプトオーバーライド付きの高速LLMを使用します。
  - 完全なフロントエンド：ノードコンポーネント、設定パネル、パレットエントリ（QU用のAIカテゴリ、DE用のDataカテゴリ）、変数ピッカーサポート。
- **フェーズ2 ワークフロー ノード：ListOperation & Transform** — 2つのデータ処理ノードタイプ：
  - **ListOperation**：8つのリスト操作（filter、map、sort、slice、flatten、unique、reverse、length）とfilter/map/sort用のsimpleeval式。JSON文字列入力の自動解析をサポートします。
  - **Transform**：6つの操作タイプ（json_path、type_cast、format、regex_extract、string_op、math_op）を備えたパイプラインベースのデータ変換。操作は順序に適用されます — 1つの出力が次の入力になります。
  - 完全なフロントエンド：ノードコンポーネント、操作固有のエディタ付き設定パネル、パレットエントリ、変数ピッカーサポート。
- **フェーズ2 ワークフロー ノード：Loop & ParameterExtractor** — 2つのより高度なノードタイプ：
  - **Loop**：simpleeval式評価、設定可能なループ変数名、max_iterations安全制限を備えたwhile条件ループ。条件入力、変数名、反復制限設定を備えたフロントエンド。
  - **ParameterExtractor**：テキストからのLLMベースの構造化パラメータ抽出。型付きパラメータ（名前/タイプ/説明/必須）を定義し、入力テキスト参照を提供し、オプションで抽出指示を追加します。解析されたJSONを返します。パラメータリストエディタと抽出プロンプトテキストエリア付きのフロントエンド。
- **フェーズ2 ワークフロー ノード：Iterator & VariableAggregator** — 2つの新しい高度なノードタイプ：
  - **Iterator**：設定可能なitem/indexの変数名とmax_iterations安全制限を備えたリストの反復。ネイティブリスト、JSON文字列解析、テンプレート変数参照をサポートします。
  - **VariableAggregator**：4つのモード経由で複数の上流ブランチからの出力をマージ — リストに収集、文字列を連結（カスタム区切り文字）、辞書をマージ、または最初の空でない値を取得。

### 変更

- **サブスクリプションベースの可視性モデル** — `build_visibility_filter()` が 3 段階 (own → org-member → subscribed) から 2 段階 (own → subscribed) に簡略化されました。データ移行により、既存のすべての組織メンバーに対して `ResourceSubscription` 行が自動作成され、アクセスが保持されます。ダッシュボードのコネクタ数と KB 統計にはサブスクライブされたリソースが含まれるようになりました。
- **Market API スコープとカテゴリパラメータ** — `browse_market()` は `scope` (market または org:`{id}`) と `category` (solutions または components) クエリパラメータを受け入れるようになりました。`subscribe_resource()` は非マーケット組織サブスクリプションの組織メンバーシップを検証するようになりました。

- **ツールキャッシュホワイトリストリファクタリング** — `ToolCache` のハードコードされたブラックリストをホワイトリストアプローチに置き換えました。`Tool` プロトコルに `cacheable` プロパティを追加 (デフォルト `False`); `cacheable=True` と明示的にマークされたツールのみがキャッシュされます。11 個の読み取り専用ツール (web_search、kb_retrieve、calculator など) がキャッシュ可能としてマークされ、すべての副作用ツール (コード実行、メール、コネクタ) はデフォルトでキャッシュされません。新しいコネクタツールは追加の設定なしで安全です。
- **DAG エグゼキュータカスケード失敗** — 失敗またはタイムアウトしたステップは、その依存ステップをカスケードブロックするようになりました。`failed_ids` トラッキングが推移的伝播 (A→B→C チェーン) で追加されました。依存ステップは、汚染されたコンテキストで実行される代わりに、説明的なメッセージとともに自動的に失敗としてマークされます。
- **DAG ステップ検証ステータス修正** — ステップ検証再試行の失敗は、`"completed"` のままにするのではなく、ステップを `"failed"` として正しくマークするようになりました。失敗した検証は、新しいカスケード失敗メカニズムを介してダウンストリームステップにカスケードされます。
- **DAG プランナーツール説明** — プランナーは、名前だけでなく各ツールの `(name, description)` を受け取るようになり、より正確なステップ-ツール マッピングが可能になり、ハルシネーション `tool_hint` 値が削減されます。
- **DAG 再プランナー完全履歴** — 再計画コンテキストは、すべての前のラウンドの結果を累積するようになりました (最後のラウンドだけではなく)。古いラウンドはより積極的に切り詰められ (200 文字/ステップ)、最新のラウンドは 500 文字/ステップを保持します。
- **stream_answer 合成品質** — 最終回答を合成する際のツール観察切り詰めが 2000 から 8000 文字に増加され、より多くの構造化データ (JSON テーブル、コード出力) が保持されます。
- **14 個のエンジン定数のパラメータ化** — ハードコードされた数値定数が ReAct エージェント、DAG プランナー、コンテキストガード、ワークスペース、およびシステムモジュール全体の環境変数に抽出されました。すべての新しい環境変数は `example.env` と `docs/configuration/environment-variables.mdx` に記載されています。

- **エクスポート: RFC 5987 ファイル名サポート**: Content-Disposition ヘッダーでプレーン `filename` より UTF-8 エンコードされた `filename*` を優先し、会話をエクスポートする際の CJK ファイル名の文字化けを修正
- **Docker i18n 破損**: `messages/` ディレクトリをスタンドアロン出力にコピーし、`outputFileTracingIncludes` を追加して、next-intl がランタイムにロケール JSON ファイルを検出できるようにした
- **Docker スタートアップ競合状態**: Next.js 起動前にエントリーポイントで API 準備完了待機ループ (Python ソケットポーリング) を追加し、`e.map is not a function` SSR エラーを防止
- **アーティファクト: 会話の created_at でソート**: ファイル mtime ではなく会話作成時刻でアーティファクトをソート (mtime は Docker ボリュームで信頼性が低い)
- **OAuth: window.location ポートから本番 API URL を自動検出**: ブラウザ自動検出が本番環境のカスタムポート (3000 だけでなく) で正しく機能するようになった
- **Docker: DooD サンドボックスボリュームマウントパスの不一致**: `CODE_EXEC_BACKEND=docker` (Docker-outside-of-Docker) でコンテナ内で実行する場合、サンドボックススクリプトはホスト Docker デーモンに見えないコンテナ内部パスに書き込まれ、空の `-v` マウントと「No such file or directory」エラーが発生していた。サンドボックスディレクトリを `data/sandbox/` に移動し、`_resolve_host_path()` によるパス変換を行う `DOCKER_HOST_DATA_DIR` 環境変数を追加し、空文字列環境変数のエッジケースを処理することで修正
