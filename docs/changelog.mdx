- **Workflow test suite expanded to 105 tests** — Added template validation tests, import/export round-trip, variable store advanced methods, deadlock detection, multi-condition branching, question classifier, and comprehensive blueprint validation. Total: 68 → 97 → 105.
- **Workflow test suite expanded to 68 tests** — Added 24 new tests covering eval namespace flattening (17 tests for dotted key resolution, short alias generation, ConditionBranch/VariableAssign with flat names) and blueprint validation warnings (7 tests for disconnected nodes, unreachable ends, empty conditions/prompts/code). Total workflow test count: 34 → 68.

- **Workflow Node Notes** — Per-node annotation feature. Click "Note" in any node's config panel to add a freeform note displayed as italic text below the node on the canvas. Auto-expanding textarea with `data.note` storage.
- **Workflow Snap-to-Grid** — 16px snap grid matching the dot background pattern for precise node alignment.
- **Workflow Connection Validation** — Prevents self-connections and duplicate edges in addition to existing Start/End restrictions.
- **Workflow Zoom-Aware Drop** — Node palette drag-and-drop uses `screenToFlowPosition()` for correct placement at any zoom level.
- **Workflow SSE Multi-Line Data** — SSE parser now handles multi-line `data:` fields and ignores keepalive comment lines.
- **Workflow Keyboard Shortcuts** — Copy (Cmd+C), paste (Cmd+V), duplicate (Cmd+D), select all (Cmd+A), Escape to deselect. Copy/paste generates new IDs and offsets positions; Start/End nodes are excluded from copy to maintain singleton constraint.
- **Workflow Inline Description Editing** — Click the description line in the toolbar to open a popover textarea for editing workflow descriptions without navigating away.
- **Workflow ConditionBranch LLM Mode** — Proper LLM-based condition evaluation replacing the naive "match first" stub. Builds a classification prompt from condition labels, calls the fast LLM, and uses case-insensitive + fuzzy matching.
- **Reactive Edge Labels** — Edge labels for condition/classifier branches now use `useNodesData()` for real-time updates when conditions or classes are edited in the config panel.
- **Workflow ENV Variables Dialog** — Per-workflow encrypted environment variable management UI. Key-value editor with masked display, duplicate key detection, and inline validation. Connected to `GET/PUT /api/workflows/{id}/env` endpoints. Accessible via toolbar dropdown.

- **Workflow List Search & Filter** — Client-side search by name/description, status filter buttons (All/Draft/Active), sort dropdown (Recently updated/created/Name A-Z), and empty state for no results.
- **Workflow Color-Coded MiniMap** — MiniMap node colors now match each node type's category color (green=Start, red=End, blue=LLM, orange=Condition, etc.).
- **Agent Skill System** — On-demand skill loading for agent instructions: `Skill` model (name, description, content, script) with full CRUD, publish review, and visibility lifecycle; `read_skill(name)` builtin tool for progressive disclosure (~80% token reduction); skill stubs injected into system prompt (~15 tokens/skill); `compact_instructions` per-agent field for custom ContextGuard compaction strategy.
- **Skills UI** — Skill list page, full-page editor (content/script/settings tabs), SkillCard component, sidebar navigation, agent editor skill selector with compact instructions textarea.
- **Workflow Publish Flow** — Publish/unpublish/resubmit endpoints for workflows, matching the existing Agent publish pattern. New `review_workflows` flag on Organization model (with Alembic migration) to enable org-level workflow review gating. Publish actions wired into workflow-card and toolbar UI.
- **Workflow Blueprint System** — Visual workflow editor for designing and executing multi-step automation blueprints:
  - **12 Node Types**: Start, End, LLM, ConditionBranch, QuestionClassifier, Agent, KnowledgeRetrieval, Connector, HTTPRequest, VariableAssign, TemplateTransform, CodeExecution — each with a custom React Flow node component, type-specific icon, and dedicated config panel form.
  - **Execution Engine**: `WorkflowEngine` with topological sort for node execution order, `asyncio.Semaphore`-based concurrency control, and condition/classifier branching that activates/deactivates downstream edges based on `active_handles`.
  - **SSE Streaming**: Real-time workflow execution progress via Server-Sent Events — emits `run_started`, `node_started`, `node_completed`, `node_failed`, `node_skipped`, `run_completed`, and `run_failed` events. SSE keepalive for long-running workflows; client disconnect detection triggers cancellation.
  - **Variable Store**: Thread-safe `VariableStore` with `{{node_id.output}}` interpolation, flat-name fallback resolution, `env.*` namespace for encrypted environment variables, `snapshot_safe()` (excludes secrets), and `list_available_variables()` for the frontend picker.
  - **Edge Labels**: `WorkflowEdgeDef` carries `source_handle` and `target_handle` for condition/classifier branches, enabling labeled edges that visually indicate which branch path each connection represents.
  - **Variable Picker Combobox**: Searchable combobox (`Command`-based popover) for inserting `{{node_id.output}}` references from upstream nodes — grouped by node type with icons, integrated into all prompt/template/body fields.
  - **Run History Sheet**: Side sheet displaying past workflow execution runs with status, duration, input/output data, and per-node results. Paginated via `GET /api/workflows/{id}/runs`.
  - **Workflow Duplicate**: `POST /api/workflows/{id}/duplicate` endpoint and UI action to clone an existing workflow (copies blueprint, schema, description; resets status to draft).
  - **Blueprint Validation**: `POST /api/workflows/validate` endpoint — validates a blueprint without saving (checks Start/End node presence, duplicate IDs, unknown node types, edge references, cycle detection). Returns node/edge counts on success or specific validation errors.
  - **Error Strategies**: Per-node `error_strategy` field with three modes — `STOP_WORKFLOW` (default: halt all remaining nodes), `CONTINUE` (mark failed but let downstream proceed), `FAIL_BRANCH` (skip all nodes reachable only through the failed node).
  - **Per-Node Timeout**: Configurable `timeout_ms` per node (default 30s) enforced via `asyncio.wait_for`; timeout produces a structured `FAILED` result with a descriptive error message.
  - **Node Config Panel**: Per-type configuration forms — LLM nodes have system prompt, model selector, model tier (fast/main) toggle, temperature slider, max tokens; ConditionBranch has operator dropdowns (==, !=, >, <, contains, etc.) with expression/LLM mode; HTTPRequest has method selector, URL, headers editor (add/remove key-value pairs), body with variable insertion; CodeExecution has language selector (Python/JavaScript) with syntax-appropriate hints; all output nodes include `OutputVariableField` and `InsertVariableBar`.
  - **Add-Node-on-Edge**: Custom `AddNodeEdge` component with "+" midpoint button — clicking opens a node type picker popup that inserts a new node inline, splitting the existing edge into two connections.
  - **Auto-Layout (ELK.js)**: `getAutoLayoutedNodes()` utility using ELK.js layered algorithm with `elk.direction: RIGHT` for horizontal left-to-right node arrangement. Configurable node spacing (60px between nodes, 120px between layers).
  - **Comprehensive Test Suite**: 34 tests covering parser validation (Start/End presence, duplicate IDs, unknown types, cycle detection, error strategy/timeout parsing), topological sort (linear and parallel branches), `VariableStore` (get/set, interpolation, env injection, snapshot safety, node outputs), engine execution (Start-to-End, input passthrough, cancellation, STOP_WORKFLOW, CONTINUE strategy), and node executors (VariableAssign, CodeExecution with variables, error handling).
  - **Backend CRUD + API**: `Workflow` and `WorkflowRun` ORM models with Alembic migration; Pydantic schemas; full REST API (`/api/workflows/*`) with create, read, update, delete, list, import/export; encrypted environment variables via `CREDENTIAL_ENCRYPTION_KEY`; variable introspection endpoint (`GET /api/workflows/{id}/variables`) for frontend config panels.
  - **Security**: Code execution node uses `subprocess` instead of bare `exec()` for OS-level isolation; environment variables filtered from eval/template execution contexts; `ExecutionContext` wiring across all node executors.
  - **Frontend**: Visual editor built with React Flow v12 — drag-and-drop node palette with category grouping (collapsible icons-only mode), 12 custom node components, run panel with SSE-driven live execution status, workflow list page with card grid. Full i18n support (6 languages).
- **Workflow Blueprint Validation Warnings**: `validate_blueprint()` now returns non-fatal warnings in addition to errors — detects disconnected nodes, unreachable end nodes, empty conditions/prompts/code blocks. Enhanced `POST /api/workflows/validate` endpoint returns structured warnings alongside validation results. 7 new tests.

### Fixed

- **Workflow eval namespace flattening**: Fixed ConditionBranch and VariableAssign failing to resolve short variable names (e.g. `score` when the store had `input.score`). Added `_flatten_eval_names()` helper that creates short aliases in the eval namespace, enabling dotted key resolution without full-path references.
- **Workflow `run_started` SSE event**: The `run_started` event was documented in the SSE protocol but never actually emitted. Now yielded before the orchestrator loop begins, so frontend consumers receive the execution-start signal.
- **Workflow SSE event name mismatch**: Fixed SSE event names — `node_status` renamed to `node_started`/`node_completed`/`node_failed`/`node_skipped` to match the frontend SSE handler expectations.
- **Workflow `NodeRunResult.duration` field name**: Fixed `duration` to `duration_ms` to match the Pydantic schema and frontend parsing.
- **Workflow export/import API response wrapping**: Fixed response envelope wrapping for workflow export and import endpoints.
- **No re-review on `is_active` toggle**: Toggling `is_active` on agents, connectors, knowledge bases, or MCP servers no longer reverts `publish_status` from `approved` back to `pending_review`. The `check_edit_revert` guard now only fires when actual content fields change (not just `is_active`).

### Changed

- **Compact Workflow Node UI**: Reduced node width from 240px to 200px, thinner color bar, smaller icons and handles for a denser visual layout. Capped `fitView` maxZoom to 1 to prevent oversized nodes on small canvases.
- **Visibility Simplified**: Resource visibility reduced from `personal | org | global` to `personal | org`. "Global" resources are now published to the Platform org, maintaining backwards-compatible behavior while simplifying the visibility model.
- **Publish Interface Unified**: All 4 resource types (Agent, Connector, Knowledge Base, MCP Server) now only support `scope="org"` for publishing. No more `scope="global"` or admin-only global publishing endpoints.
- **Auto-Platform Enrollment**: New users are automatically added to the Platform org on registration, ensuring immediate access to shared organizational resources.
- **Admin Panel Cleanup**: Removed "Global Agents" and "Global MCP" admin tabs, as global resources are now managed through the Platform org.

### Removed

- Removed `is_global` field from Agent and MCPServer models and all API responses
- Removed `cloned_from_agent_id` and `cloned_from_mcp_server_id` fields from Agent and MCPServer
- Removed global agent management endpoints (`/api/admin/global-agents/*`)
- Removed global MCP server endpoints (`/api/admin/mcp-servers/*`)
- Removed all `visibility="global"` concept references from resources, API contracts, and frontend logic

## [v0.7.5] - 2026-03-12

### Added
- **Free mode switching + per-turn mode tracking**: Users can now freely switch between Auto/React/DAG mid-conversation without starting a new chat. Each assistant message stores its actual execution mode in `metadata.mode`. History turns are rendered with the correct engine per-turn (detected from metadata or SSE event fingerprinting). Sidebar mode icons removed. Per-turn mode badge shows which engine processed each turn.
- **Three model roles (F5)**: Independent environment config for General, Fast, and Reasoning model tiers. Fast model no longer inherits reasoning settings from the main model. New env vars: `FAST_LLM_API_KEY`, `FAST_LLM_BASE_URL`, `FAST_LLM_TEMPERATURE`, `REASONING_LLM_MODEL`, `REASONING_LLM_API_KEY`, `REASONING_LLM_BASE_URL`, `REASONING_LLM_TEMPERATURE`, `REASONING_LLM_CONTEXT_SIZE`, `REASONING_LLM_MAX_OUTPUT_TOKENS`, `REASONING_LLM_EFFORT`, `REASONING_LLM_BUDGET`
- **Structured step output (F2)**: `StepOutput` dataclass replaces plain string results in DAG steps, enabling future support for structured data and artifact passing between steps
- **Tool cache (F3)**: Identical tool calls within a single DAG execution are cached to avoid redundant API calls. Per-key async locks prevent thundering herd. Configurable via `DAG_TOOL_CACHE` (default: on)
- **Per-step verification (F4)**: Optional LLM-based quality check after each DAG step completes. On failure, retries once with verification feedback. Configurable via `DAG_STEP_VERIFICATION` (default: off)
- **Auto-routing (F1)**: Fast LLM classifies queries as ReAct or DAG execution mode. New `/api/auto` endpoint with SSE routing event. Frontend 3-way mode toggle (Auto/Standard/Planner). Configurable via `AUTO_ROUTING` (default: on)
- **Design philosophy doc**: New `docs/design-philosophy.mdx` documenting architectural decisions — why Handoff, Backflow, Teams, and agent_hint were intentionally excluded from DAG execution

- **Connector Credentials Encryption + Per-User Override** — Auth tokens (`bearer`, `api_key`, `basic`) are now extracted from `auth_config` and stored separately in a new `connector_credentials` table with optional Fernet encryption (set `CREDENTIAL_ENCRYPTION_KEY`). Sensitive fields are stripped from all API responses. New endpoints: `GET/PUT/DELETE /api/connectors/{id}/my-credentials` allow users to store personal credential overrides. `allow_fallback` column on connectors controls whether personal credential absence falls back to the connector owner's default. Chat tool loading resolves per-user credentials first, then falls back to owner credential when `allow_fallback=true`. Existing credentials are automatically migrated from `auth_config` by migration `s9u1w3y5a678`.
- **ModelConfig API key encryption at rest** — `ModelConfig.api_key` is now encrypted at rest using the `EncryptedString` TypeDecorator (Fernet symmetric encryption via `CREDENTIAL_ENCRYPTION_KEY`). Transparent encrypt-on-write / decrypt-on-read with backward-compatible legacy plaintext detection. Migration `f8g9h0a1b2c3` converts existing plaintext values in-place.

### Fixed
- Fast model (`get_fast_llm()`) no longer inherits `LLM_API_KEY`, `LLM_BASE_URL`, `LLM_TEMPERATURE`, and reasoning settings from the main model configuration
- Backend Pydantic schemas now accept `"auto"` mode for conversations and agent `execution_mode`, with default changed from `"react"` to `"auto"`
- Frontend SSE routing event field names aligned with backend (`chosen_mode`→`mode`, `reason`→`reasoning`) — auto-routing was silently falling back to ReAct regardless of classifier decision
- Agent settings form default execution mode changed from `"react"` to `"auto"` to match backend schema defaults

### Changed
- Mode toggle labels now include engine name: "Standard (ReAct)", "Planner (DAG)" for clarity
- **Skeleton screens** — All list/grid pages now show layout-aware skeleton placeholders during initial load instead of a centered spinner. Affected pages: `/agents` (3-col `AgentCard` × 6), `/chats` (`ListRow` × 8), `/kb` (3-col `KbCard` × 6), `/kb/[id]` (structural skeleton with header + toolbar + `TableRow` × 5), `/connectors` connectors tab (3-col `ConnectorCard` × 6), `/artifacts` (4-col `ArtifactCard` × 8). Three new composites added to `skeleton.tsx`: `ConnectorCard`, `ArtifactCard`, `TableRow`. Tab-switch and secondary loading states (show-more, search, dialogs) retain Loader2 spinners.

## [v0.7.4] - 2026-03-12

- **Evaluation Center** — New `/eval` section in the portal for quantitative agent quality benchmarking. Features: Test Dataset management (create/edit collections of prompt + expected behavior + assertion test cases); Eval Runs (select an agent + dataset → run all cases in parallel through the live agent with `asyncio.Semaphore(5)` → grade each answer with a fast LLM grader → persist per-case pass/fail + latency + token results); Results viewer with expandable rows, pass-rate progress bar, and auto-polling (3s) while running. Suitable for enterprise procurement review. Migration `r8t0v2x4z567`.
- **Admin: `json_mode_enabled` per-model flag** — Each model config now has an explicit JSON Mode toggle (default on) in Admin → Models → Advanced settings. Disabling it prevents LiteLLM from injecting an assistant prefill message (`{"role":"assistant","content":"{"}`) that causes AWS Bedrock to reject conversations with a `ValidationException` on the 2nd+ agent iteration. ENV-configured models are controlled by the new `LLM_JSON_MODE_ENABLED` env var. All URL-pattern heuristics (`is_bedrock`, `is_anthropic_relay`) have been removed; capability is now explicit. Includes Alembic migration (`q7s9u1w3y456`).

Features include a Create Organization dialog (name, auto-generated slug, description, emoji icon), an Edit dialog (owner only), a Members Sheet for managing member roles (owner/admin/member with color-coded badges), and inline actions (leave for non-owners, delete for owners). All org operations call the user-facing `/api/orgs` endpoints. Full i18n support (EN + ZH).

SSE protocol emits `done` immediately after the answer, then sends `suggestions`, `title`, and `end` as separate events. DB session early-release during LLM execution. New `is_builder` boolean column on agents with Alembic migration. Builder prompt auto-refresh on session creation. SSRF guard for OpenAPI import

CSS dot-pulse streaming cursor. DAG layout decoupled from step states. Collapsed sidebar new-chat button. Connector editor overflow fix. `font-mono` replaced with `tabular-nums`
- **AI Builder concept page**: New `docs/concepts/ai-builder.mdx` with connector and agent builder guides, plus Chinese translation

An "Advanced Builder" button opens a full-screen dialog with the embedded PlaygroundPage (chat on the left, live action preview on the right). Six connector builder tools (`connector_list_actions`, `connector_create_action`, `connector_update_action`, `connector_delete_action`, `connector_update_settings`, `connector_test_action`) operate directly on the DB and are auto-injected when the Builder Agent is active. `POST /api/builder/session` creates or reuses a per-connector builder agent that persists conversation history

SQLite remains the zero-config default; PostgreSQL is recommended for production and multi-worker deployments. Docker Compose auto-provisions a PostgreSQL service with health checks. Alembic migrations are dialect-aware (json_extract vs `::json->>'key'`). Admin health dashboard shows the actual database engine and version
