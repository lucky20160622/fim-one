All notable changes to this project are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/). Versions correspond to [Roadmap](/roadmap) milestones.

## [Unreleased]

### Added

- **Per-model Native Function Calling toggle** — New `tool_choice_enabled` setting (ENV global + Admin per-model) controls whether forced `tool_choice` is attempted for structured output extraction. Models that reject forced tool selection (e.g. thinking-mode models) can skip Level 1 and go directly to JSON Mode, eliminating the ~10s penalty per structured call. Configurable in Settings → Models → Advanced.
- **Structured Output Compatibility table** in model documentation — Shows which providers support native function calling, JSON mode, and planning reliability ratings.
- **File attachment download** — File attachment cards in chat messages are now clickable to download the original file via authenticated fetch. Download icon indicator and hover state added for discoverability.
- **Intelligent Document Processing (Vision-Aware) roadmap item** — Added planned feature for adaptive document handling based on model vision capabilities (three tiers: vision-first, hybrid, text-only).

### Changed

- **Temperature hint moved to tooltip** — The temperature reasoning hint in the model form dialog is now displayed as a compact tooltip icon (HelpCircle) next to the label instead of a static paragraph below the slider, reducing visual clutter.
- **RAG grounding pipeline** — Three-stage pipeline: retrieval, citation extraction, and confidence scoring. Confidence score is the average retrieval score of the top-3 evidence units.
- **API Keys banners updated to "Active"** — Admin and user settings API Keys pages now show a green "API Keys Active" banner (replacing the previous "Coming Soon" notice) with a link to the API authentication docs.
- **Multi-worker documentation clarified** — `REDIS_URL` and `WORKERS` environment variable descriptions updated with a warning checklist covering stop, inject, PostgreSQL requirements, and local dev guidance.

### Fixed

- **Duplicate OpenAPI Endpoints in locale docs** — Removed redundant `Endpoints` OpenAPI group from all non-English locale API tabs in `docs.json`. The English section retains the single source; locale sections no longer duplicate it.
- **Structured output degradation chain** — The 3-level fallback (native FC → JSON mode → plain text extraction) now properly falls through all levels when a model produces malformed JSON. Previously short-circuited at Level 2 with a `ValueError`.
- **`json_mode_enabled` DB value ignored** — Models configured via Admin (model groups) now correctly use their per-model `json_mode_enabled` setting instead of always falling back to the `LLM_JSON_MODE_ENABLED` environment variable.
- **DAG planning failure message** — Planning failures now show a user-friendly bilingual message suggesting model switch instead of a raw pipeline error.
- **DAG step card layout** — Redesigned `StepProgressCard` so task text wraps below the step badge instead of truncating on a single line. Tool badges in step nodes now only display when no `tools_used` exist but `tool_hint` is present, avoiding redundant display.
- **MCP server owner bypass for allow_fallback** — Server owner is no longer blocked by `allow_fallback=False` restriction. Previously, even the creator of an MCP server could be skipped if `allow_fallback=False` was set; now only non-owner users without credentials are excluded.
- **Model provider API key validation** — Creating a non-openai-compatible provider now requires an API key with inline validation error, required asterisk indicator, and proper `aria-invalid` accessibility attribute.

### Changed

- **Subscription-based visibility model** — `build_visibility_filter()` simplified from 3-tier (own → org-member → subscribed) to 2-tier (own → subscribed). Data migration auto-creates `ResourceSubscription` rows for all existing org members to preserve access. Dashboard connector count and KB stats now include subscribed resources.
- **Market API scope and category params** — `browse_market()` accepts `scope` (market or org:`{id}`) and `category` (solutions or components) query params. `subscribe_resource()` now validates org membership for non-market org subscriptions.

- **Tool cache whitelist refactor** — Replaced hardcoded blacklist in `ToolCache` with a whitelist approach. Added `cacheable` property to `Tool` protocol (default `False`); only tools explicitly marked `cacheable=True` are cached. 11 read-only tools (web_search, kb_retrieve, calculator, etc.) marked cacheable; all side-effect tools (code exec, email, connectors) remain uncached by default. New connector tools are safe without any additional configuration.
- **DAG executor cascade failure** — Failed or timed-out steps now cascade-block their dependents. Added `failed_ids` tracking with transitive propagation (A→B→C chains). Dependent steps are automatically marked as failed with a descriptive message instead of running with poisoned context.
- **DAG step verifier status fix** — Step verification retry failures now correctly mark the step as `"failed"` instead of leaving it as `"completed"`. Failed verifications cascade to downstream steps via the new cascade failure mechanism.
- **DAG planner tool descriptions** — Planner now receives `{name, description}` for each tool instead of just names, enabling more accurate step-to-tool mapping and reducing hallucinated `tool_hint` values.
- **DAG re-planner full history** — Re-planning context now accumulates results from ALL previous rounds (not just the last one). Older rounds are more aggressively truncated (200 chars/step) while the most recent round retains 500 chars/step.
- **stream_answer synthesis quality** — Increased tool observation truncation from 2000 to 8000 characters when synthesizing final answers, preserving more structured data (JSON tables, code output).
- **14 engine constants parameterized** — Extracted hardcoded numeric constants to environment variables across ReAct agent, DAG planner, context guard, workspace, and system modules. All new env vars documented in `example.env` and `docs/configuration/environment-variables.mdx`.

- **Evidence confidence UI enhancements** — Three improvements to the grounding pipeline's frontend display: (1) insufficient-evidence observations now render as i18n amber warning cards instead of raw English text, (2) inline `[N]` citation badges show hover Popovers with source name, quote excerpt, and relevance score via a new EvidenceContext, (3) conflict warning banner appears in the References section when evidence sources contradict each other, with expandable side-by-side comparison.
- **Shadow Market org** — Transformed the built-in Platform org into a shadow Market org. Resources are now discovered via marketplace browsing and explicitly subscribed to (pull model) instead of auto-visible via membership (push model). Non-admin users no longer auto-join any org on registration; the Market org operates as an invisible backend entity that users interact with only through the marketplace UI.
- **Marketplace publish with mandatory review** — Publishing to the Market org now always requires review, except for Market org owner/admins. The publish dialog shows a "Publish to Marketplace" option with a mandatory review notice.
- **Skill and Workflow marketplace browsing** — Added Skill and Workflow resource types to the marketplace, enabling community discovery and subscription for all five resource types (agents, connectors, knowledge bases, MCP servers, skills/workflows).
- **Market org publish endpoint bypass** — Six resource publish endpoints now skip membership checks for the Market org, allowing any user to publish resources to the marketplace without being a member.
- **Platform-to-Market migration** — Alembic migration renames the Platform org to Market and removes all non-owner memberships, completing the transition from push-based to pull-based resource sharing.
- **Workflow version change summaries** — `compute_blueprint_diff()` auto-generates human-readable summaries from blueprint diffs when saving new versions, describing added/removed/modified nodes and edges.
- **Workflow run retention cleanup** — Background cleanup task that purges old workflow runs based on configurable age and count limits. Per-workflow retention overrides. Admin endpoint for manual cleanup trigger. New env vars: `WORKFLOW_RUN_MAX_AGE_DAYS`, `WORKFLOW_RUN_MAX_PER_WORKFLOW`, `WORKFLOW_RUN_CLEANUP_INTERVAL_HOURS`.
- **Connector circuit breaker** — Three-state circuit breaker (closed/open/half-open) with per-connector failure tracking. Detects 5xx errors and temporarily disables failing connectors. Monitoring endpoints for circuit breaker state inspection.
- **ConnectorMetaTool (Progressive Disclosure Phase 1-2)** — Single meta-tool replaces per-action tools in the agent prompt. System prompt receives lightweight stubs (~30 tokens/connector vs ~250 tokens/action); agent calls `discover(connector)` to load full action schema on demand and `execute` to run actions. Feature flag `CONNECTOR_TOOL_MODE` for backward compatibility.
- **Real MCP executor** — Replaced stub MCP node executor with full implementation supporting MCP server discovery and tool calling within workflow execution.
- **Agent mid-loop self-reflection** — Lightweight goal-check prompt injected every 6 tool-call iterations in the ReAct loop (both JSON and native modes) to prevent goal drift in long reasoning chains. Asks the agent whether it is still on track, repeating actions, or should produce a final answer.

Each field can carry `semantic_tag`, `description`, and `pii` flags, surfaced in LLM tool descriptions so the agent understands field intent without guessing from column names.
