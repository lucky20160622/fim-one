## [Unreleased]

### Added

- **Agent Skill System** — On-demand skill loading for agent instructions: `Skill` model (name, description, content, script) with full CRUD, publish review, and visibility lifecycle; `read_skill(name)` builtin tool for progressive disclosure (~80% token reduction); skill stubs injected into system prompt (~15 tokens/skill); `compact_instructions` per-agent field for custom ContextGuard compaction strategy.
- **Skills UI** — Skill list page, full-page editor (content/script/settings tabs), SkillCard component, sidebar navigation, agent editor skill selector with compact instructions textarea.
- **Workflow Publish Flow** — Publish/unpublish/resubmit endpoints for workflows, matching the existing Agent publish pattern. New `review_workflows` flag on Organization model (with Alembic migration) to enable org-level workflow review gating. Publish actions wired into workflow-card and toolbar UI.
- **Workflow Blueprint System** — Visual workflow editor for designing and executing multi-step automation blueprints. Full-stack implementation:
  - **Backend**: `Workflow` and `WorkflowRun` ORM models with Alembic migration; Pydantic schemas for CRUD + validation; full REST API (`/api/workflows/*`) with create, read, update, delete, list, import/export; SSE-based execution endpoint for real-time run status streaming; encrypted environment variables via existing `CREDENTIAL_ENCRYPTION_KEY`.
  - **Execution Engine**: `WorkflowEngine` with topological sort for node execution order, condition branching support, and `VariableStore` for cross-node data passing. 12 node executors: Start, End, LLM, ConditionBranch, QuestionClassifier, Agent, KnowledgeRetrieval, Connector, HTTPRequest, VariableAssign, TemplateTransform, CodeExecution.
  - **Frontend**: Visual workflow editor built with React Flow v12 — drag-and-drop node palette, 12 custom node components with type-specific icons and configuration, node config panel with per-type forms, connection validation, run panel with SSE-driven live execution status, workflow list page with card grid. Full i18n support (6 languages).
  - **Security**: Code execution node uses `subprocess` instead of bare `exec()` for OS-level isolation; environment variables filtered from eval/template execution contexts; SSE keepalive for long-running workflows; proper `ExecutionContext` wiring across all node executors.
- **Admin: Review Log audit tab** — Admin audit page now has a segmented "System Log / Review Log" toggle. The Review Log panel queries the new `GET /api/admin/review-log` endpoint (filters: org, resource type, action; limit/offset pagination) and displays a table with time, org name, resource type, resource name, action badge (color-coded: green=approved, red=rejected, amber=submitted/resubmitted, muted=unpublished), and operator. Full i18n (EN + ZH).
- **Platform Organization**: Built-in org `platform` (ID: `00000000-0000-0000-0000-000000000001`) — all new users automatically join. This organization replaces the "global" visibility concept and provides a central hub for sharing resources across all users.
- **Resource Subscriptions**: Users can now subscribe/unsubscribe to shared resources (agents, connectors, knowledge bases, MCP servers) from org Markets. New `resource_subscriptions` database table tracks subscription relationships.
- **Market API**: New endpoints for browsing and managing subscriptions:
  - `GET /api/market` — browse available resources in current org
  - `POST /api/market/subscribe` — subscribe to a resource
  - `DELETE /api/market/unsubscribe` — unsubscribe from a resource
  - `GET /api/market/subscriptions` — list user's current subscriptions
- **Market Page**: New `/market` page in Portal for browsing and managing subscriptions to org resources.
- **Agent Auto-discovery**: New `discoverable` boolean field on agents. Agents with `discoverable=true` can be automatically discovered and called by LLMs via the `call_agent` tool (one-level delegation, no recursion).
- **Sub-agent Binding**: New `sub_agent_ids` field on agents — explicit whitelist of callable sub-agents for fine-grained access control.
- **CallAgentTool**: Built-in tool that delegates tasks to specialist agents with structured input validation and result handling.
- **MCP Server Credentials**: New `mcp_server_credentials` table enables per-user environment variable and HTTP header overrides for MCP servers.
- **MCP Credentials Endpoint**: `PUT /api/mcp-servers/{id}/my-credentials` allows users to store personal credential overrides for shared MCP servers.
- **`allow_fallback` on MCPServer**: Boolean field controlling credential fallback behavior — when `true`, missing user credentials fall back to owner defaults.
- **Connector/KB Toggle**: New endpoints for resource state management:
  - `POST /api/connectors/{id}/toggle` — suspend/resume connector
  - `POST /api/knowledge-bases/{id}/toggle` — suspend/resume knowledge base
- **Standalone KB Conversations**: New `kb_ids` field on `ChatStreamRequest` and `Conversation` — users can now chat directly with knowledge bases without binding them to an agent.
- **Platform Org Protection**: The Platform org cannot be deleted, and users cannot self-leave it. Ensures all users maintain connectivity to the organization hub.
- **Publish Review UI (Frontend)**: Org-level publish review system frontend — review toggle in org edit dialog, ReviewsSheet with type/status filters and approve/reject workflow, publish status badges (pending/approved/rejected) on agent cards with tooltip for rejection notes, review notice in publish dialog when target org requires review, resubmit action for rejected resources, "Review Management" dropdown for org admin/owners. All resource types (agents, connectors, KBs, MCP servers) gain `publish_status`, `reviewed_by`, `reviewed_at`, `review_note` type fields. Full i18n (EN + ZH).

### Fixed

- **Workflow SSE event name mismatch**: Fixed SSE event names — `node_status` renamed to `node_started`/`node_completed`/`node_failed`/`node_skipped` to match the frontend SSE handler expectations.
- **Workflow `NodeRunResult.duration` field name**: Fixed `duration` to `duration_ms` to match the Pydantic schema and frontend parsing.
- **Workflow export/import API response wrapping**: Fixed response envelope wrapping for workflow export and import endpoints.
- **No re-review on `is_active` toggle**: Toggling `is_active` on agents, connectors, knowledge bases, or MCP servers no longer reverts `publish_status` from `approved` back to `pending_review`. The `check_edit_revert` guard now only fires when actual content fields change (not just `is_active`).

### Changed

- **Compact Workflow Node UI**: Reduced node width from 240px to 200px, thinner color bar, smaller icons and handles for a denser visual layout. Capped `fitView` maxZoom to 1 to prevent oversized nodes on small canvases.
- **Visibility Simplified**: Resource visibility reduced from `personal | org | global` to `personal | org`. "Global" resources are now published to the Platform org, maintaining backwards-compatible behavior while simplifying the visibility model.
- **Publish Interface Unified**: All 4 resource types (Agent, Connector, Knowledge Base, MCP Server) now only support `scope="org"` for publishing. No more `scope="global"` or admin-only global publishing endpoints.
- **Auto-Platform Enrollment**: New users are automatically added to the Platform org on registration, ensuring immediate access to shared organizational resources.
- **Admin Panel Cleanup**: Removed "Global Agents" and "Global MCP" admin tabs, as global resources are now managed through the Platform org.

### Removed

- Removed `is_global` field from Agent and MCPServer models and all API responses
- Removed `cloned_from_agent_id` and `cloned_from_mcp_server_id` fields from Agent and MCPServer
- Removed global agent management endpoints (`/api/admin/global-agents/*`)
- Removed global MCP server endpoints (`/api/admin/mcp-servers/*`)
- Removed all `visibility="global"` concept references from resources, API contracts, and frontend logic

## [v0.7.5] - 2026-03-12

### Added
- **Free mode switching + per-turn mode tracking**: Users can now freely switch between Auto/React/DAG mid-conversation without starting a new chat. Each assistant message stores its actual execution mode in `metadata.mode`. History turns are rendered with the correct engine per-turn (detected from metadata or SSE event fingerprinting). Sidebar mode icons removed. Per-turn mode badge shows which engine processed each turn.
- **Three model roles (F5)**: Independent environment config for General, Fast, and Reasoning model tiers. Fast model no longer inherits reasoning settings from the main model. New env vars: `FAST_LLM_API_KEY`, `FAST_LLM_BASE_URL`, `FAST_LLM_TEMPERATURE`, `REASONING_LLM_MODEL`, `REASONING_LLM_API_KEY`, `REASONING_LLM_BASE_URL`, `REASONING_LLM_TEMPERATURE`, `REASONING_LLM_CONTEXT_SIZE`, `REASONING_LLM_MAX_OUTPUT_TOKENS`, `REASONING_LLM_EFFORT`, `REASONING_LLM_BUDGET`
- **Structured step output (F2)**: `StepOutput` dataclass replaces plain string results in DAG steps, enabling future support for structured data and artifact passing between steps
- **Tool cache (F3)**: Identical tool calls within a single DAG execution are cached to avoid redundant API calls. Per-key async locks prevent thundering herd. Configurable via `DAG_TOOL_CACHE` (default: on)
- **Per-step verification (F4)**: Optional LLM-based quality check after each DAG step completes. On failure, retries once with verification feedback. Configurable via `DAG_STEP_VERIFICATION` (default: off)
- **Auto-routing (F1)**: Fast LLM classifies queries as ReAct or DAG execution mode. New `/api/auto` endpoint with SSE routing event. Frontend 3-way mode toggle (Auto/Standard/Planner). Configurable via `AUTO_ROUTING` (default: on)
- **Design philosophy doc**: New `docs/design-philosophy.mdx` documenting architectural decisions — why Handoff, Backflow, Teams, and agent_hint were intentionally excluded from DAG execution

- **Connector Credentials Encryption + Per-User Override** — Auth tokens (`bearer`, `api_key`, `basic`) are now extracted from `auth_config` and stored separately in a new `connector_credentials` table with optional Fernet encryption (set `CREDENTIAL_ENCRYPTION_KEY`). Sensitive fields are stripped from all API responses. New endpoints: `GET/PUT/DELETE /api/connectors/{id}/my-credentials` allow users to store personal credential overrides. `allow_fallback` column on connectors controls whether personal credential absence falls back to the connector owner's default. Chat tool loading resolves per-user credentials first, then falls back to owner credential when `allow_fallback=true`. Existing credentials are automatically migrated from `auth_config` by migration `s9u1w3y5a678`.
- **ModelConfig API key encryption at rest** — `ModelConfig.api_key` is now encrypted at rest using the `EncryptedString` TypeDecorator (Fernet symmetric encryption via `CREDENTIAL_ENCRYPTION_KEY`). Transparent encrypt-on-write / decrypt-on-read with backward-compatible legacy plaintext detection. Migration `f8g9h0a1b2c3` converts existing plaintext values in-place.

### Fixed
- Fast model (`get_fast_llm()`) no longer inherits `LLM_API_KEY`, `LLM_BASE_URL`, `LLM_TEMPERATURE`, and reasoning settings from the main model configuration
- Backend Pydantic schemas now accept `"auto"` mode for conversations and agent `execution_mode`, with default changed from `"react"` to `"auto"`
- Frontend SSE routing event field names aligned with backend (`chosen_mode`→`mode`, `reason`→`reasoning`) — auto-routing was silently falling back to ReAct regardless of classifier decision
- Agent settings form default execution mode changed from `"react"` to `"auto"` to match backend schema defaults

### Changed
- Mode toggle labels now include engine name: "Standard (ReAct)", "Planner (DAG)" for clarity
- **Skeleton screens** — All list/grid pages now show layout-aware skeleton placeholders during initial load instead of a centered spinner. Affected pages: `/agents` (3-col `AgentCard` × 6), `/chats` (`ListRow` × 8), `/kb` (3-col `KbCard` × 6), `/kb/[id]` (structural skeleton with header + toolbar + `TableRow` × 5), `/connectors` connectors tab (3-col `ConnectorCard` × 6), `/artifacts` (4-col `ArtifactCard` × 8). Three new composites added to `skeleton.tsx`: `ConnectorCard`, `ArtifactCard`, `TableRow`. Tab-switch and secondary loading states (show-more, search, dialogs) retain Loader2 spinners.

## [v0.7.4] - 2026-03-12

### Added
- **Evaluation Center** — New `/eval` section in the portal for quantitative agent quality benchmarking. Features: Test Dataset management (create/edit collections of prompt + expected behavior + assertion test cases); Eval Runs (select an agent + dataset → run all cases in parallel through the live agent with `asyncio.Semaphore(5)` → grade each answer with a fast LLM grader → persist per-case pass/fail + latency + token results); Results viewer with expandable rows, pass-rate progress bar, and auto-polling (3s) while running. Suitable for enterprise procurement review. Migration `r8t0v2x4z567`.
- **Admin: `json_mode_enabled` per-model flag** — Each model config now has an explicit JSON Mode toggle (default on) in Admin → Models → Advanced settings. Disabling it prevents LiteLLM from injecting an assistant prefill message (`{"role":"assistant","content":"{"}`) that causes AWS Bedrock to reject conversations with a `ValidationException` on the 2nd+ agent iteration. ENV-configured models are controlled by the new `LLM_JSON_MODE_ENABLED` env var. All URL-pattern heuristics (`is_bedrock`, `is_anthropic_relay`) have been removed; capability is now explicit. Includes Alembic migration (`q7s9u1w3y456`).
- **Admin model form UX** — Restructured Add/Edit Model dialog: only Name, Model Name, and API Key are shown by default. All other settings (Base URL, Provider, Temperature, Max Output Tokens, Context Size, JSON Mode) are collapsed under an "Advanced" toggle, reducing visual noise for the common case.
- **Settings: Organizations tab** — Users can now create, join, and manage organizations directly from the Settings page. Features include a Create Organization dialog (name, auto-generated slug, description, emoji icon), an Edit dialog (owner only), a Members Sheet for managing member roles (owner/admin/member with color-coded badges), and inline actions (leave for non-owners, delete for owners). All org operations call the user-facing `/api/orgs` endpoints. Full i18n support (EN + ZH).
- **SSE Protocol v2 — real-time answer streaming**: Backend `StreamChunk` gains `delta_reasoning` and `usage` fields; LLM requests include usage tracking; ReAct `_run_native()` now streams via `stream_chat`. SSE protocol emits `done` immediately after the answer, then sends `suggestions`, `title`, and `end` as separate events. DB session early-release during LLM execution. SQLite pool size increased from 5 to 20
- **AI Builder expansion — 7 new tools + `is_builder` flag**: Three new connector builder tools (`GetSettings`, `TestConnection`, `ImportOpenAPI`) and four new agent builder tools (`ListConnectors`, `AddConnector`, `RemoveConnector`, `SetModel`). New `is_builder` boolean column on agents with Alembic migration. Builder prompt auto-refresh on session creation. SSRF guard for OpenAPI import
- **SSE v2 frontend — streaming cursor, DAG round snapshots, UI polish**: Frontend hooks parse `end` / `suggestions` / `title` events. DAG re-plan shows previous failed rounds as collapsible cards. CSS dot-pulse streaming cursor. DAG layout decoupled from step states. Collapsed sidebar new-chat button. Connector editor overflow fix. `font-mono` replaced with `tabular-nums`
- **AI Builder concept page**: New `docs/concepts/ai-builder.mdx` with connector and agent builder guides, plus Chinese translation
- **Builder Dialog — ReAct agent for connector management**: Upgrades the AI assistant panel from a single-shot LLM call to a full ReAct agent loop. An "Advanced Builder" button opens a full-screen dialog with the embedded PlaygroundPage (chat on the left, live action preview on the right). Six connector builder tools (`connector_list_actions`, `connector_create_action`, `connector_update_action`, `connector_delete_action`, `connector_update_settings`, `connector_test_action`) operate directly on the DB and are auto-injected when the Builder Agent is active. `POST /api/builder/session` creates or reuses a per-connector builder agent that persists conversation history
- **Dual database support (SQLite + PostgreSQL)**: `DATABASE_URL` now accepts both SQLite (`sqlite+aiosqlite:///...`) and PostgreSQL (`postgresql+asyncpg://...`) connection strings. SQLite remains the zero-config default; PostgreSQL is recommended for production and multi-worker deployments. Docker Compose auto-provisions a PostgreSQL service with health checks. Alembic migrations are dialect-aware (json_extract vs `::json->>'key'`). Admin health dashboard shows the actual database engine and version
- **Models configuration docs**: New `docs/configuration/models.mdx` page documenting model configuration reference, including extended thinking / reasoning setup for each provider
- **Extended thinking / reasoning support**: Add `LLM_REASONING_EFFORT` (low/medium/high) and `LLM_REASONING_BUDGET_TOKENS` env vars for models with extended thinking (OpenAI o-series, Gemini 2.5+, Claude). Sends `reasoning_effort` for OpenAI/Gemini, `thinking` for Anthropic (auto-detected). Extracts `reasoning_content` from multiple provider field names (DeepSeek R1, Claude proxies) and surfaces it in the UI "thinking" step
- **Admin: tool disable**: New admin Tools panel with per-tool enable/disable toggles (stored as `disabled_builtin_tools` setting). Disabled tools are filtered from chat tool resolution at runtime. The `/api/tools/catalog` endpoint returns a `disabled` flag, and the Tools page UI shows disabled state
- **Admin: MCP servers on Connectors page**: MCP servers management moved from Tools page to Connectors page as a new tab. `MCPServersSection` extracted into a reusable component with imperative action refs
- **Admin: reasoning effort in health dashboard**: Reasoning effort label now displayed in the admin health check dashboard
- **Export: PDF format**: Conversations can now be exported as PDF documents (requires `reportlab` + `Markdown` extras)
- **start.sh: WORKERS env var**: Set `WORKERS=N` to run uvicorn with multiple workers in portal/api mode

### Changed

- **LLM layer**: Replaced direct `AsyncOpenAI` client with [LiteLLM](https://github.com/BerriAI/litellm) for universal provider support.
