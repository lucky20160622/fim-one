All configuration is done via `.env`. Copy `example.env` and fill in your values:

```bash
cp example.env .env
```

## Configuration Levels

Each integration has a configuration level indicating its importance:

| Level | Meaning | Behavior when not configured |
|-------|---------|------------------------------|
| **Required** | Core system dependency | System will error â€” chat and primary functions will not work |
| **Recommended** | Significant feature enabler | Graceful degradation â€” the feature is visibly unavailable but the system runs |
| **Optional** | Enhancement capability | Transparent degradation â€” system works fine, capability simply not present |

> **Note**: Admin-configured models (Admin â†’ Models page) can substitute for LLM environment variables. The health check considers both sources.

## Frontend (Local Dev Only)

The frontend has a separate env file **only for local development**: `frontend/.env.local`.

> **This file is NOT used in Docker.** Inside the Docker container, Next.js proxies `/api/*` to the Python backend internally (port 8000 is container-internal), so no frontend env file is needed.

For local dev, the defaults work out of the box â€” you do **not** need to create `frontend/.env.local` unless your backend runs on a non-default port.

If you need to override, create `frontend/.env.local` manually:

```bash
echo 'NEXT_PUBLIC_API_URL=http://localhost:9000' > frontend/.env.local
```

| Variable | Default | Description |
|---|---|---|
| `NEXT_PUBLIC_API_URL` | `http://localhost:8000` *(auto)* | Backend URL the **browser** uses for direct API calls (OAuth redirects, streaming). Auto-detected from `window.location` if unset â€” only override if your backend runs on a non-standard port locally. |

> **Build-time note**: `NEXT_PUBLIC_*` variables are baked into the JS bundle at `pnpm build` time. Changing them at runtime (e.g. via root `.env`) has no effect â€” this is why they live in `frontend/.env.local` for local dev only.

### Extended Thinking (Reasoning)

When `LLM_REASONING_EFFORT` is set, FIM Agent enables the model's extended thinking capability so the internal chain-of-thought is surfaced in the UI "thinking" step. **We currently support two official provider paths only:**

#### Supported providers

| Provider | `LLM_BASE_URL` | API parameter sent | Reasoning content returned? |
|---|---|---|---|
| **OpenAI** (o1 / o3 / o4-mini) | `https://api.openai.com/v1` | `reasoning_effort: "high"` | Yes |
| **Anthropic** (Claude 3.7+) | `https://api.anthropic.com/v1/` | `extra_body: {thinking: {type: "enabled", budget_tokens: N}}` | No (compat endpoint limitation) |

The system auto-detects Anthropic by checking whether `LLM_BASE_URL` contains `anthropic.com`.

- **Google Gemini** also supports `reasoning_effort` on its OpenAI-compatible endpoint (`generativelanguage.googleapis.com/v1beta/openai/`), so it should work out of the box â€” but we have not officially tested it.

#### Important caveats

<Warning>
**Anthropic's OpenAI-compatible endpoint does not return reasoning content.**
If you set `LLM_BASE_URL` to Anthropic's official endpoint (`https://api.anthropic.com/v1/`), the `thinking` parameter will be sent correctly and the model *will* think internally, but the response **will not include the reasoning text** â€” you will only see the final answer. The UI "thinking" step will appear but with empty reasoning content. This is a limitation of Anthropic's compatibility layer, not a bug.

For full reasoning visibility with Claude, use the native Anthropic SDK or a third-party proxy that returns `reasoning_content`.
</Warning>

<Warning>
**Third-party proxies / custom endpoints are not guaranteed.**
If your `LLM_BASE_URL` points to a third-party API proxy (e.g., OpenRouter, one-api, custom gateway), the `reasoning_effort` parameter is sent as-is. Whether the proxy translates it correctly to the underlying provider depends entirely on the proxy's implementation. Some proxies expect a different format (e.g., `reasoning: {effort, max_tokens}`). If reasoning does not work through your proxy, consult the proxy's documentation for their expected parameter format â€” FIM Agent cannot adapt to arbitrary proxy conventions.
</Warning>

#### How `LLM_REASONING_BUDGET_TOKENS` works

This variable is **only meaningful for the Anthropic path**. When set, it overrides the auto-calculated budget and is sent as `budget_tokens` in the `thinking` parameter. When not set, the budget is derived from `LLM_MAX_OUTPUT_TOKENS` Ă— effort ratio:

| `LLM_REASONING_EFFORT` | Budget ratio | Example (max_tokens = 64000) |
|---|---|---|
| `low` | 20% | 12,800 tokens |
| `medium` | 50% | 32,000 tokens |
| `high` | 80% | 51,200 tokens |

The minimum budget is 1,024 tokens (Anthropic's hard minimum).

For OpenAI and Gemini, the provider handles token allocation internally based on the `reasoning_effort` level â€” `LLM_REASONING_BUDGET_TOKENS` has no effect.
