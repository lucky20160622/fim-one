Tous les changements notables de ce projet sont documentés ici.

Le format suit [Keep a Changelog](https://keepachangelog.com/). Les versions correspondent aux jalons de la [Roadmap](/roadmap).

## [Non publié]

### Ajouté

- **Téléchargement de pièces jointes** — Les cartes de pièces jointes dans les messages de chat sont maintenant cliquables pour télécharger le fichier original via une récupération authentifiée. Un indicateur d'icône de téléchargement et un état de survol ont été ajoutés pour la découverte.
- **Traitement intelligent des documents (Vision-Aware) - élément de feuille de route** — Ajout d'une fonctionnalité prévue pour la gestion adaptative des documents en fonction des capacités de vision du modèle (trois niveaux : vision-first, hybride, texte uniquement).

### Modifié

- **Pipeline d'ancrage RAG** — Pipeline en trois étapes : récupération, extraction de citations et notation de confiance. Le score de confiance est la moyenne des scores de récupération des 3 meilleures unités de preuve.

### Corrigé

- **Disposition de la carte d'étape DAG** — Redessiné `StepProgressCard` pour que le texte de la tâche s'enroule sous le badge d'étape au lieu de se tronquer sur une seule ligne. Les badges d'outil dans les nœuds d'étape ne s'affichent désormais que lorsqu'aucun `tools_used` n'existe mais que `tool_hint` est présent, évitant un affichage redondant.

## [v0.8] - 2026-03-20

### Ajouté

- **Directive d'agent dans la synthèse ReAct** — Nouveau paramètre `agent_directive` sur `ReActAgent` injecte l'objectif principal de l'agent (par exemple « traduire le chinois en anglais ») dans le message système de synthèse, garantissant que la réponse finale respecte l'identité de l'agent au lieu de recourir par défaut à des réponses conversationnelles. Câblé dans les deux points de terminaison de chat ReAct et DAG.

- **Refonte de la Marketplace Phase 1 — Modèle à deux niveaux Solutions + Composants** — Remplacé l'onglet Market plat à 6 types par une navigation structurée à deux niveaux : bascule de catégorie de haut niveau (Solutions : Agent/Compétence/Flux de travail ; Composants : Connecteur/Serveur MCP) et sous-onglets de type. Ajout d'une liste déroulante de sélecteur de portée pour basculer entre le Marché Global et les organisations de l'utilisateur. KB supprimée de la portée du Marché (org uniquement). Partage unifié org et marché : les deux nécessitent maintenant un abonnement explicite (plus d'apparition automatique org).

- **Injection intelligente de contenu de fichier** — Les petits fichiers téléchargés (<32K caractères) sont automatiquement récupérés et intégrés au contexte LLM ; les gros fichiers reçoivent un résumé des métadonnées avec un conseil d'outil pour une lecture à la demande.
- **Outil d'agent `read_uploaded_file`** — Outil de lecture de fichier en mode dual : lecture paginée (offset/limit) pour l'accès séquentiel et recherche de type grep (requête avec support regex) pour l'extraction de contenu ciblée.
- **Point de terminaison `GET /api/files/{file_id}/content`** — API de contenu texte paginée pour accéder au texte complet extrait des fichiers téléchargés.
- **Fichiers sidecar `.content`** — Texte complet extrait stocké aux côtés des téléchargements en tant que fichiers sidecar, permettant l'accès au contenu persistant sans réextraction.
- **Champ `content_length` dans les réponses de l'API de fichiers** — Les points de terminaison de téléchargement et de liste de fichiers incluent maintenant `content_length` pour la visibilité de la taille du contenu texte.
- **Méthode client frontend `fileApi.getContent()`** — Nouvelle méthode client API pour récupérer le contenu de fichier paginé depuis le frontend.

- **Nouvelle tentative de nœud du moteur de flux de travail** — Nouvelle tentative par nœud avec backoff exponentiel. Les nœuds peuvent spécifier `retry_count` et `retry_delay_ms` dans leur configuration de données. Les nœuds défaillants sont relancés jusqu'à N fois avec un délai plafonné à 60s. Émet des événements SSE `node_retrying` avec le numéro de tentative et l'erreur précédente. Respecte les événements d'annulation pendant l'attente de nouvelle tentative.
- **Évaluation d'expression du magasin de variables de flux de travail** — Évaluation d'expression Python sécurisée via simpleeval. La méthode `evaluate_expression()` expose les variables de magasin (excluant env) avec les builtins courants (len, str, int, float, bool, abs, min, max, sum, round, sorted). Plus les méthodes d'aide `has()`, `delete()`, `keys()`.
- **Nœud Phase 2 du flux de travail : HumanIntervention** — Nœud pause-pour-approbation (stub d'approbation automatique). Lit prompt_message (avec interpolation {{variable}}), assignee, timeout_hours. Nœud frontend à thème Sky-500 avec aperçu d'invite et badge @assignee. Panneau de configuration avec textarea d'invite, entrée assignee, champ timeout.
- **Modèles de flux de travail étendus à 12** — Quatre nouveaux modèles : Pipeline d'approbation humaine (LLM → HumanIntervention → ConditionBranch → End dual), Traitement de données multi-étapes (CodeExecution → ConditionBranch → Transform → ListOperation), Agent amélioré par les connaissances (QuestionUnderstanding → KnowledgeRetrieval → LLM), Boucle d'examen itérative (Loop → LLM → VariableAssign).
- **Menu contextuel du volet de flux de travail** — Clic droit sur le canevas vide pour Coller (désactivé quand le presse-papiers est vide), Sélectionner tout, Ajuster la vue et Actions d'auto-disposition. L'état d'union discriminée sépare les menus contextuels de nœud et de volet.
- **Menu contextuel du nœud de flux de travail** — Clic droit sur n'importe quel nœud pour accéder aux actions Configurer, Copier, Dupliquer et Supprimer. L'action Supprimer est masquée pour les nœuds singleton Start/End. Le menu se ferme automatiquement au clic extérieur ou au défilement.
- **Modèles Phase 2 du flux de travail** — Deux nouveaux modèles intégrés (8 au total) : Pipeline de traitement de liste (Start → ListOperation → Transform → End) et Réponse aux questions améliorée (Start → QuestionUnderstanding → LLM → End).
- **Validation Phase 2 du flux de travail** — Avertissements de validation de blueprint étendus pour ListOperation (entrée/expression manquante), Transform (entrée/opérations manquantes/vides), Iterator (variable de liste manquante), Loop (condition manquante), DocumentExtractor, QuestionUnderstanding (entrée manquante), ParameterExtractor (texte d'entrée/paramètres manquants/vides). Corrigé : l'opération de tri ne nécessite plus incorrectement une expression.
- **Notes de nœud de flux de travail** — Fonction d'annotation par nœud. Cliquez sur « Note » dans le panneau de configuration de n'importe quel nœud pour ajouter une note en texte libre affichée en italique sous le nœud sur le canevas. Textarea à expansion automatique avec stockage `data.note`.
- **Magnétisme à la grille du flux de travail** — Grille d'accrochage 16px correspondant au motif de fond pointillé pour un alignement précis des nœuds.
- **Validation de connexion du flux de travail** — Empêche les auto-connexions et les arêtes dupliquées en plus des restrictions Start/End existantes.
- **Dépôt sensible au zoom du flux de travail** — Le glisser-déposer de la palette de nœuds utilise `screenToFlowPosition()` pour un placement correct à n'importe quel niveau de zoom.
- **Données multi-lignes SSE du flux de travail** — L'analyseur SSE gère maintenant les champs `data:` multi-lignes et ignore les lignes de commentaire keepalive.
- **Raccourcis clavier du flux de travail** — Copier (Cmd+C), coller (Cmd+V), dupliquer (Cmd+D), sélectionner tout (Cmd+A), Échap pour désélectionner. Copier/coller génère de nouveaux ID et décale les positions ; les nœuds Start/End sont exclus de la copie pour maintenir la contrainte singleton.
- **Édition de description en ligne du flux de travail** — Cliquez sur la ligne de description dans la barre d'outils pour ouvrir un textarea popover pour éditer les descriptions de flux de travail sans vous éloigner.
- **Mode LLM ConditionBranch du flux de travail** — Évaluation de condition appropriée basée sur LLM remplaçant le stub naïf « correspondance en premier ». Construit une invite de classification à partir des étiquettes de condition, appelle le LLM rapide et utilise la correspondance insensible à la casse + floue.
- **Étiquettes d'arête réactives** — Les étiquettes d'arête pour les branches condition/classificateur utilisent maintenant `useNodesData()` pour les mises à jour en temps réel lorsque les conditions ou les classes sont modifiées dans le panneau de configuration.
- **Panneau de statistiques du flux de travail** — Affichage de statistiques basé sur feuille montrant les exécutions totales, les nombres complétés/échoués/annulés, le taux de réussite avec barre de progression visuelle, la durée moyenne et l'heure de la dernière exécution avec formatage relatif. Déclenché à partir d'une icône BarChart3 dans la barre d'outils.
- **Nœuds Phase 2 du flux de travail : DocumentExtractor & QuestionUnderstanding** — Deux autres types de nœuds :
  - **DocumentExtractor** : Exécuteur stub pour le traitement de texte de document — prend en charge les types d'entrée texte/base64 et 4 modes d'extraction (full_text, pages avec page_range, métadonnées avec comptes de mots/caractères/lignes/pages, tableaux avec détection de tableau markdown). L'entrée URL est reportée pour une implémentation ultérieure.
  - **QuestionUnderstanding** : Prétraitement de question basé sur LLM avec 4 modes — réécrire (clarté), développer (contexte + sous-questions), classer (JSON intention/sujet/confiance), décomposer (tableau de sous-questions). Utilise le LLM rapide avec remplacement de message système personnalisé optionnel.
  - Frontend complet : composants de nœud, panneaux de configuration, entrées de palette (catégorie AI pour QU, catégorie Data pour DE), support du sélecteur de variables.
- **Nœuds Phase 2 du flux de travail : ListOperation & Transform** — Deux types de nœuds de traitement de données :
  - **ListOperation** : 8 opérations de liste (filter, map, sort, slice, flatten, unique, reverse, length) avec expressions simpleeval pour filter/map/sort. Prend en charge l'analyse automatique d'entrée de chaîne JSON.
  - **Transform** : Transformation de données basée sur pipeline avec 6 types d'opération (json_path, type_cast, format, regex_extract, string_op, math_op). Les opérations sont appliquées séquentiellement — la sortie de l'une alimente la suivante.
  - Frontend complet : composants de nœud, panneau de configuration avec éditeurs spécifiques à l'opération, entrées de palette, support du sélecteur de variables.
- **Nœuds Phase 2 du flux de travail : Loop & ParameterExtractor** — Deux autres types de nœuds avancés :
  - **Loop** : Boucle while-condition avec évaluation d'expression simpleeval, nom de variable de boucle configurable et limite de sécurité max_iterations. Frontend avec entrée de condition, nom de variable et configuration de limite d'itération.
  - **ParameterExtractor** : Extraction de paramètres structurés basée sur LLM à partir du texte. Définissez des paramètres typés (nom/type/description/requis), fournissez une référence de texte d'entrée et ajoutez optionnellement des instructions d'extraction. Retourne JSON analysé. Frontend avec éditeur de liste de paramètres et textarea d'invite d'extraction.
- **Nœuds Phase 2 du flux de travail : Iterator & VariableAggregator** — Deux nouveaux types de nœuds avancés :
  - **Iterator** : Itérer sur les listes avec noms de variables d'élément/index configurables et limite de sécurité max_iterations. Prend en charge les listes natives, l'analyse de chaîne JSON et les références de variables de modèle.
  - **VariableAggregator** : Fusionner les sorties de plusieurs branches en amont via quatre modes — collecter dans une liste, concaténer des chaînes (séparateur personnalisé), fusionner des dicts ou prendre la première valeur non vide.
  - Intégration frontend complète : composants de nœud, entrées de palette, formulaires de panneau de configuration, couleurs de minimap, support du sélecteur de variables. Exécuteurs backend enregistrés dans le moteur.
- **Modèles de flux de travail étendus** — Ajout de deux nouveaux modèles intégrés (6 au total) : Pipeline de traitement de données (Start → CodeExecution → TemplateTransform → End) et Agent avec récupération de connaissances (Start → KnowledgeRetrieval → Agent → End). Correction des champs `icon` manquants sur les nouveaux modèles. Ajout des clés i18n manquantes pour les catégories de modèles « intermédiaire » et « avancé ».
- **Dialogue des variables ENV du flux de travail** — Gestion de l'interface utilisateur des variables d'environnement chiffrées par flux de travail. Éditeur clé-valeur avec affichage masqué, détection de clé dupliquée et validation en ligne. Connecté aux points de terminaison `GET/PUT /api/workflows/{id}/env`. Accessible via la liste déroulante de la barre d'outils.
- **Sauvegarde automatique du flux de travail** — Minuteur de sauvegarde automatique débattu de 5 secondes avec remplacement immédiat Cmd+S. L'indicateur d'état de sauvegarde affiche « Enregistrement... », « Modifications non enregistrées » ou « Enregistré il y a Xs » avec actualisation automatique de 30s.
- **Recherche et filtrage de la liste de flux de travail** — Recherche côté client par nom/description, boutons de filtre d'état (Tous/Brouillon/Actif), liste déroulante de tri (Récemment mis à jour/créé/Nom A-Z) et état vide pour aucun résultat.
- **MiniMap codé par couleur du flux de travail** — Les couleurs des nœuds MiniMap correspondent maintenant à la couleur de catégorie de chaque type de nœud (vert=Start, rouge=End, bleu=LLM, orange=Condition, etc.).
- **Système de compétences d'agent** — Chargement de compétences à la demande pour les instructions d'agent : modèle `Skill` (nom, description, contenu, script) avec CRUD complet, examen de publication et cycle de vie de visibilité ; outil intégré `read_skill(name)` pour la divulgation progressive (~80% réduction de jetons) ; stubs de compétences injectés dans le message système (~15 jetons/compétence) ; champ `compact_instructions` par agent pour la stratégie de compaction ContextGuard personnalisée.
- **Interface utilisateur des compétences** — Page de liste de compétences, éditeur pleine page (onglets contenu/script/paramètres), composant SkillCard, navigation de barre latérale, sélecteur de compétences d'éditeur d'agent avec textarea d'instructions compactes.
- **Flux de publication du flux de travail** — Points de terminaison de publication/dépublication/renvoi pour les flux de travail, correspondant au modèle de publication d'agent existant

### Modifié

- **Modèle de visibilité basé sur les abonnements** — `build_visibility_filter()` simplifié de 3 niveaux (propre → membre de l'org → abonné) à 2 niveaux (propre → abonné). La migration des données crée automatiquement des lignes `ResourceSubscription` pour tous les membres de l'org existants afin de préserver l'accès. Le nombre de connecteurs du tableau de bord et les statistiques de KB incluent désormais les ressources abonnées.
- **Paramètres de portée et de catégorie de l'API Market** — `browse_market()` accepte les paramètres de requête `scope` (market ou org:{id}) et `category` (solutions ou components). `subscribe_resource()` valide désormais l'appartenance à l'org pour les abonnements org non-market.

- **Refactorisation de la liste blanche du cache des outils** — Remplacement de la liste noire codée en dur dans `ToolCache` par une approche de liste blanche. Ajout de la propriété `cacheable` au protocole `Tool` (par défaut `False`) ; seuls les outils explicitement marqués `cacheable=True` sont mis en cache. 11 outils en lecture seule (web_search, kb_retrieve, calculator, etc.) marqués comme cacheable ; tous les outils avec effets secondaires (code exec, email, connecteurs) restent non mis en cache par défaut. Les nouveaux outils de connecteur sont sûrs sans configuration supplémentaire.
- **Défaillance en cascade de l'exécuteur DAG** — Les étapes échouées ou expirées bloquent désormais en cascade leurs dépendances. Ajout du suivi `failed_ids` avec propagation transitive (chaînes A→B→C). Les étapes dépendantes sont automatiquement marquées comme échouées avec un message descriptif au lieu de s'exécuter avec un contexte empoisonné.
- **Correction du statut du vérificateur d'étape DAG** — Les défaillances de nouvelle tentative de vérification d'étape marquent désormais correctement l'étape comme `"failed"` au lieu de la laisser comme `"completed"`. Les vérifications échouées se propagent aux étapes en aval via le nouveau mécanisme de défaillance en cascade.
- **Descriptions des outils du planificateur DAG** — Le planificateur reçoit désormais `{name, description}` pour chaque outil au lieu de simplement les noms, permettant un mappage étape-vers-outil plus précis et réduisant les valeurs `tool_hint` halluccinées.
- **Historique complet du re-planificateur DAG** — Le contexte de re-planification accumule désormais les résultats de TOUS les tours précédents (pas seulement le dernier). Les tours plus anciens sont tronqués plus agressivement (200 caractères/étape) tandis que le tour le plus récent conserve 500 caractères/étape.
- **Qualité de synthèse de stream_answer** — Augmentation de la troncature d'observation des outils de 2000 à 8000 caractères lors de la synthèse des réponses finales, préservant plus de données structurées (tableaux JSON, sortie de code).
- **14 constantes du moteur paramétrées** — Extraction des constantes numériques codées en dur vers des variables d'environnement dans l'agent ReAct, le planificateur DAG, le garde de contexte, l'espace de travail et les modules système. Toutes les nouvelles variables d'env documentées dans `example.env` et `docs/configuration/environment-variables.mdx`.

- **Améliorations de l'interface utilisateur de confiance des preuves** — Trois améliorations du pipeline de fondation de l'affichage frontal : (1) les observations de preuves insuffisantes s'affichent désormais sous forme de cartes d'avertissement ambre i18n au lieu de texte anglais brut, (2) les badges de citation en ligne `[N]` affichent des Popovers au survol avec le nom de la source, l'extrait de citation et le score de pertinence via un nouveau EvidenceContext, (3) la bannière d'avertissement de conflit apparaît dans la section Références lorsque les sources de preuves se contredisent, avec comparaison côte à côte extensible.
- **Org Market fantôme** — Transformation de l'org Platform intégré en org Market fantôme. Les ressources sont désormais découvertes via la navigation du marché et explicitement abonnées (modèle pull) au lieu d'être automatiquement visibles via l'appartenance (modèle push). Les utilisateurs non-administrateurs ne rejoignent plus automatiquement aucune org lors de l'inscription ; l'org Market fonctionne comme une entité backend invisible avec laquelle les utilisateurs n'interagissent que via l'interface du marché.
- **Publication sur le marché avec révision obligatoire** — La publication sur l'org Market nécessite désormais toujours une révision, sauf pour le propriétaire/les administrateurs de l'org Market. La boîte de dialogue de publication affiche une option « Publier sur le marché » avec un avis de révision obligatoire.
- **Navigation du marché des compétences et des flux de travail** — Ajout des types de ressources Skill et Workflow au marché, permettant la découverte communautaire et l'abonnement pour les cinq types de ressources (agents, connecteurs, bases de connaissances, serveurs MCP, compétences/flux de travail).
- **Contournement du point de terminaison de publication de l'org Market** — Six points de terminaison de publication de ressources contournent désormais les vérifications d'appartenance pour l'org Market, permettant à tout utilisateur de publier des ressources sur le marché sans être membre.
- **Migration Platform vers Market** — Migration Alembic qui renomme l'org Platform en Market et supprime tous les membres non-propriétaires, complétant la transition du partage basé sur le push au partage basé sur le pull.
- **Résumés des modifications de version du flux de travail** — `compute_blueprint_diff()` génère automatiquement des résumés lisibles par l'homme à partir des diffs de blueprint lors de l'enregistrement de nouvelles versions, décrivant les nœuds et arêtes ajoutés/supprimés/modifiés.
- **Nettoyage de la rétention des exécutions de flux de travail** — Tâche de nettoyage en arrière-plan qui purge les anciennes exécutions de flux de travail en fonction des limites d'âge et de nombre configurables. Remplacements de rétention par flux de travail. Point de terminaison administrateur pour le déclenchement manuel du nettoyage. Nouvelles variables d'env : `WORKFLOW_RUN_MAX_AGE_DAYS`, `WORKFLOW_RUN_MAX_PER_WORKFLOW`, `WORKFLOW_RUN_CLEANUP_INTERVAL_HOURS`.
- **Disjoncteur de connecteur** — Disjoncteur à trois états (fermé/ouvert/semi-ouvert) avec suivi des défaillances par connecteur. Détecte les erreurs 5xx et désactive temporairement les connecteurs défaillants. Points de terminaison de surveillance pour l'inspection de l'état du disjoncteur.
- **ConnectorMetaTool (Phase de divulgation progressive 1-2)** — Un seul méta-outil remplace les outils par action dans l'invite de l'agent. L'invite système reçoit des stubs légers (~30 jetons/connecteur vs ~250 jetons/action) ; l'agent appelle `discover(connector)` pour charger le schéma d'action complet à la demande et `execute` pour exécuter les actions. Drapeau de fonctionnalité `CONNECTOR_TOOL_MODE` pour la compatibilité rétroactive.
- **Exécuteur MCP réel** — Remplacement de l'exécuteur de nœud MCP stub par une implémentation complète supportant la découverte de serveur MCP et l'appel d'outils lors de l'exécution du flux de travail.
- **Auto-réflexion à mi-boucle de l'agent** — Invite de vérification d'objectif léger injectée tous les 6 appels d'outils dans la boucle ReAct (modes JSON et natif) pour prévenir la dérive d'objectif dans les longues chaînes de raisonnement. Demande à l'agent s'il est toujours sur la bonne voie, répète les actions ou doit produire une réponse finale.

- **Exécuteur BuiltinTool réel** — Remplacement de l'exécuteur de nœud BuiltinTool stub par une implémentation réelle utilisant l'intégration `ToolRegistry` pour l'exécution du flux de travail.
- **Annotations de schéma sémantique** — 16 balises sémantiques prédéfinies (`customer_id`, `email`, `phone`, `amount`, `date`, etc.) pour les champs de schéma de connecteur. Chaque champ peut porter les drapeaux `semantic_tag`, `description` et `pii`, affichés dans les descriptions d'outils LLM afin que l'agent comprenne l'intention du champ sans deviner à partir des noms de colonnes.
- **Interface utilisateur d'import/export/fork de connecteur** — Actions de carte frontale pour exporter les connecteurs en JSON, importer à partir du sélecteur de fichiers et forker (cloner + personnaliser) les connecteurs existants.
- **Points de terminaison d'import/export et de fork de connecteur** — API backend : `GET /api/connectors/{id}/export`, `POST /api/connectors/import`, `POST /api/connectors/{id}/fork` avec gestion des identifiants sanitisée pour prévenir la fuite de secrets.
- **Système de modèles de flux de travail** — Modèle ORM `WorkflowTemplate` avec CRUD administrateur et points de terminaison de listing/clonage publics. 5 modèles de départ insérés automatiquement au premier démarrage pour l'amorçage rapide du flux de travail.
- **Badges de validation en ligne du flux de travail** — `ValidationBadge` en temps réel sur les nœuds de canevas affichant les erreurs et avertissements par nœud avec des info-bulles descriptives, fournissant un retour visuel immédiat lors de l'édition du blueprint.
- **Visionneuse de trace d'exécution du flux de travail** — Visionneuse de trace basée sur une chronologie pour l'inspection des exécutions terminées. Le paramètre `trace_level` du moteur contrôle la granularité des détails ; les instantanés de variables sont capturés à chaque nœud pour le débogage pas à pas.
- **Limitation de débit par utilisateur du flux de travail et délai d'expiration global** — `WorkflowRateLimiter` avec application de fenêtre glissante (10 exécutions/min, 3 simultanées par utilisateur). Délai d'expiration global par défaut de 10 minutes (600s) appliqué à toutes les exécutions de flux de travail.
- **Gestion des flux de travail en lot administrateur** — Trois points de terminaison administrateur en lot pour les flux de travail : suppression en lot, basculement en lot (actif/inactif) et publication en lot, tous avec journalisation d'audit.
- **Améliorations de la palette de flux de travail** — Section des nœuds récemment utilisés via le hook `useRecentNodes`, support de la recherche par description et état vide « aucun résultat » dans la palette de nœuds.
- **Superposition de relecture d'exécution du flux de travail** — Bouton « Afficher sur le canevas » dans les superpositions d'historique d'exécution qui superpose les résultats d'exécution passés sur le canevas du flux de travail, affichant le statut par nœud et la sortie d'une exécution terminée sans réexécution.
- **Exécuteur SubWorkflow complet** — Mise à niveau du nœud SubWorkflow de stub à exécution imbriquée complète sauvegardée en base de données. Charge le flux de travail cible à partir de la base de données, exécute avec les variables d'entrée mappées, applique une limite de profondeur configurable pour prévenir la récursion infinie et retourne les sorties d'exécution imbriquées au flux de travail parent.
- **Onglet de gestion des flux de travail administrateur** — Nouvel onglet « Flux de travail » dans le panneau administrateur pour gérer tous les flux de travail entre les utilisateurs.
