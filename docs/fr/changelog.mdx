## [Non publié]

### Modifié

- **Troncature centralisée des outils** — Tous les types d'outils (WebFetch, HttpRequest, ShellExec, PythonExec, NodeExec) délèguent désormais la troncature au module partagé `truncation.py` au lieu d'utiliser une logique intégrée. Deux points d'entrée : `truncate_tool_output()` (compatible JSON, au niveau des caractères) pour les outils web/API, `truncate_bytes()` (au niveau des octets) pour les outils d'exécution de code. Les valeurs par défaut sont configurables via les variables d'environnement `TOOL_OUTPUT_MAX_CHARS`, `TOOL_OUTPUT_MAX_ITEMS`, `TOOL_OUTPUT_MAX_BYTES`.
- **Détection de domaine découplée en middleware indépendant** — La classification de domaine (juridique, médical, financier, etc.) est désormais un module autonome (`domain.py`) qui s'exécute indépendamment dans chaque endpoint (ReAct et DAG), au lieu d'être regroupée avec la classification du routage automatique. Les deux endpoints appellent `classify_domain()` directement, supprimant la dépendance au routeur automatique pour propager `domain_hint`.
- **Instructions SOP de domaine assouplies** — Les instructions de domaine spécialisées guident désormais plutôt que de mandater la recherche web. Lorsque la requête de l'utilisateur et le contexte contiennent déjà suffisamment d'informations, l'agent répond directement au lieu de forcer des recherches inutiles.
- **`web_search` toujours épinglé dans ReAct** — L'outil `web_search` est désormais toujours inclus dans la sélection des outils ReAct, indépendamment de la détection d'un domaine spécialisé, garantissant qu'il ne peut pas être supprimé par le sélecteur d'outils.
- **Variable d'environnement `AUTO_ROUTING` supprimée** — L'endpoint auto classe désormais toujours les requêtes ; le bouton bascule pour désactiver la classification a été supprimé.
- **Le vérificateur de citations accepte `domain_hint` en amont** — `should_verify_citations()` accepte désormais un paramètre `domain_hint` provenant de la classification LLM au niveau de la requête, contournant l'analyse par mot-clé à chaque étape lorsque le domaine est déjà connu.

### Ajouté

- **Basculement natif d'appel de fonction par modèle** — Nouveau paramètre `tool_choice_enabled` (global ENV + Admin par modèle) contrôle si `tool_choice` forcé est tenté pour l'extraction de sortie structurée. Les modèles qui rejettent la sélection d'outil forcée (par exemple, les modèles en mode réflexion) peuvent ignorer le niveau 1 et passer directement au mode JSON, éliminant la pénalité d'environ 10 secondes par appel structuré. Configurable dans Paramètres → Modèles → Avancé.
- **Tableau de compatibilité de sortie structurée** dans la documentation des modèles — Affiche les fournisseurs qui prennent en charge l'appel de fonction natif, le mode JSON et les évaluations de fiabilité de la planification.
- **Téléchargement de pièce jointe** — Les cartes de pièce jointe dans les messages de chat sont désormais cliquables pour télécharger le fichier original via une récupération authentifiée. Indicateur d'icône de téléchargement et état de survol ajoutés pour la découverte.
- **Élément de feuille de route de traitement intelligent de documents (Vision-Aware)** — Fonctionnalité planifiée pour la gestion adaptative des documents en fonction des capacités de vision du modèle (trois niveaux : vision-first, hybride, texte uniquement).
- **Révision de la qualité DAG (P0) : Mise à niveau du modèle par défaut** — Les étapes DAG sans `model_hint` explicite utilisent désormais le modèle général du registre (par exemple, Sonnet) au lieu du modèle rapide. Seules les étapes avec `model_hint="fast"` utilisent le modèle léger.
- **Révision de la qualité DAG (P1) : Découverte automatique des compétences** — `DAGPlanner.plan()` accepte un paramètre `skill_descriptions`. Les compétences disponibles (nom + description) sont injectées dans l'invite de planification afin que le planificateur puisse suggérer `tool_hint="read_skill"` pour les étapes correspondant au domaine.
- **Révision de la qualité DAG (P2) : Vérificateur de citations** — Nouveau module `citation_verifier.py` extrait les citations juridiques/médicales/financières via regex, vérifie la précision via jugement LLM et réessaie en cas d'échec avec retour de correction. Intégré dans le flux post-étape de l'exécuteur. Contrôlé par la variable d'environnement `DAG_CITATION_VERIFICATION` (par défaut : `true`).
- **Révision de la qualité DAG (P3) : Préservation du contexte de contenu structuré** — `_build_step_context()` détecte le contenu structuré (citations juridiques, tableaux markdown, blocs de code) dans les résultats de dépendance et applique un multiplicateur de troncature configurable (`DAG_STRUCTURED_CONTEXT_MULTIPLIER`, par défaut `3.0`), empêchant la perte de citations critiques entre les étapes DAG.
- **Révision de la qualité DAG (P4) : Classification de domaine dans le routeur** — `RouteDecision` inclut désormais `domain_hint` (juridique/médical/financier/null). L'analyse de domaine étroitement couplée préfère le mode ReAct ; lorsque DAG est sélectionné, le contexte de domaine est injecté dans le planificateur pour guider les choix de `model_hint`.
- **Escalade de modèle de domaine dans ReAct** — Lorsque le routeur détecte un domaine spécialisé (juridique/médical/financier), le point de terminaison ReAct passe du modèle général au modèle de raisonnement via `registry.get_by_role("reasoning")`. Les instructions obligatoires sont injectées exigeant une recherche web avant la rédaction et la vérification des citations via la recherche. L'outil `read_skill` est épinglé dans la sélection d'outils.

### Modifié

- **Indice de température déplacé vers une infobulle** — L'indice de raisonnement de température dans la boîte de dialogue du formulaire de modèle est maintenant affiché sous forme d'une icône d'infobulle compacte (HelpCircle) à côté du libellé au lieu d'un paragraphe statique sous le curseur, réduisant l'encombrement visuel.
- **Pipeline d'ancrage RAG** — Pipeline en trois étapes : récupération, extraction de citations et notation de confiance. Le score de confiance est la moyenne des scores de récupération des 3 meilleures unités de preuve.
- **Bannières de clés API mises à jour en « Actif »** — Les pages Clés API des paramètres administrateur et utilisateur affichent maintenant une bannière verte « Clés API Actif » (remplaçant l'avis « À venir » précédent) avec un lien vers la documentation d'authentification API.
- **Documentation multi-worker clarifiée** — Les descriptions des variables d'environnement `REDIS_URL` et `WORKERS` mises à jour avec une liste de contrôle d'avertissement couvrant l'arrêt, l'injection, les exigences PostgreSQL et les conseils de développement local.
- **Modèle par défaut des étapes DAG** — Les étapes sans `model_hint` se résolvent maintenant au modèle général (précédemment modèle rapide). Le modèle rapide n'est utilisé que lorsque `model_hint="fast"` est explicitement défini.
- **Sortie de classification du routeur** — `RouteDecision` génère maintenant `domain_hint` (legal/medical/financial/null) aux côtés du champ `mode` existant, permettant la sélection de modèle consciente du domaine en aval.

### Corrigé

- **Doublons des points de terminaison OpenAPI dans les documents de locale** — Suppression du groupe OpenAPI `Endpoints` redondant de tous les onglets API de locale non-anglaise dans `docs.json`. La section anglaise conserve la source unique ; les sections de locale ne la dupliquent plus.
- **Chaîne de dégradation de la sortie structurée** — Le fallback à 3 niveaux (FC natif → mode JSON → extraction de texte brut) traverse maintenant correctement tous les niveaux lorsqu'un modèle produit du JSON malformé. Auparavant, il s'arrêtait au niveau 2 avec une `ValueError`.
- **Valeur `json_mode_enabled` de la base de données ignorée** — Les modèles configurés via Admin (groupes de modèles) utilisent maintenant correctement leur paramètre `json_mode_enabled` par modèle au lieu de toujours revenir à la variable d'environnement `LLM_JSON_MODE_ENABLED`.
- **Message d'échec de planification DAG** — Les échecs de planification affichent maintenant un message bilingue convivial suggérant un changement de modèle au lieu d'une erreur de pipeline brute.
- **Disposition de la carte d'étape DAG** — Redessin de `StepProgressCard` pour que le texte de la tâche s'enroule sous le badge d'étape au lieu de se tronquer sur une seule ligne. Les badges d'outil dans les nœuds d'étape ne s'affichent maintenant que lorsqu'aucun `tools_used` n'existe mais que `tool_hint` est présent, évitant un affichage redondant.
- **Contournement du propriétaire du serveur MCP pour allow_fallback** — Le propriétaire du serveur n'est plus bloqué par la restriction `allow_fallback=False`. Auparavant, même le créateur d'un serveur MCP pouvait être ignoré si `allow_fallback=False` était défini ; maintenant, seuls les utilisateurs non-propriétaires sans identifiants sont exclus.
- **Validation de la clé API du fournisseur de modèle** — La création d'un fournisseur non compatible avec openai nécessite maintenant une clé API avec validation d'erreur en ligne, indicateur d'astérisque requis et attribut d'accessibilité `aria-invalid` approprié.

## [v0.8] - 2026-03-20

### Ajoutés

- **Directive d'agent dans la synthèse ReAct** — Nouveau paramètre `agent_directive` sur `ReActAgent` qui injecte l'objectif principal de l'agent (par ex. « traduire du chinois vers l'anglais ») dans l'invite système de synthèse, garantissant que la réponse finale respecte l'identité de l'agent au lieu de revenir par défaut à des réponses conversationnelles. Intégré dans les deux points de terminaison de chat ReAct et DAG.

- **Refonte de la Marketplace Phase 1 — Modèle à deux niveaux Solutions + Composants** — Remplacement de l'onglet Market plat à 6 types par une navigation structurée à deux niveaux : bascule de catégorie de haut niveau (Solutions : Agent/Compétence/Flux de travail ; Composants : Connecteur/Serveur MCP) et sous-onglets de type. Ajout d'une liste déroulante de sélecteur d'étendue pour basculer entre la Marketplace mondiale et les organisations de l'utilisateur. KB supprimée de l'étendue Market (org uniquement). Partage unifié org et market : les deux nécessitent désormais un abonnement explicite (plus d'apparition automatique org).

- **Injection intelligente de contenu de fichier** — Les petits fichiers téléchargés (`<32K` caractères) sont automatiquement récupérés et intégrés au contexte LLM ; les gros fichiers reçoivent un résumé de métadonnées avec un indice d'outil pour la lecture à la demande.
- **Outil d'agent `read_uploaded_file`** — Outil de lecture de fichier en mode double : lecture paginée (offset/limit) pour l'accès séquentiel et recherche de type grep (requête avec support regex) pour l'extraction de contenu ciblée.
- **Point de terminaison `GET /api/files/{file_id}/content`** — API de contenu texte paginée pour accéder au texte extrait complet des fichiers téléchargés.
- **Fichiers sidecar `.content`** — Texte extrait complet stocké aux côtés des téléchargements en tant que fichiers sidecar, permettant l'accès au contenu persistant sans réextraction.
- **Champ `content_length` dans les réponses de l'API fichier** — Les points de terminaison de téléchargement et de liste de fichiers incluent désormais `content_length` pour la visibilité de la taille du contenu texte.
- **Méthode client frontend `fileApi.getContent()`** — Nouvelle méthode client API pour récupérer le contenu de fichier paginé depuis le frontend.

- **Nouvelle tentative de nœud du moteur de flux de travail** — Nouvelle tentative par nœud avec backoff exponentiel. Les nœuds peuvent spécifier `retry_count` et `retry_delay_ms` dans leur configuration de données. Les nœuds défaillants sont retentés jusqu'à N fois avec un délai plafonné à 60s. Émet des événements SSE `node_retrying` avec le numéro de tentative et l'erreur précédente. Respecte les événements d'annulation pendant l'attente de nouvelle tentative.

La méthode `evaluate_expression()` expose les variables de magasin (excluant env) avec les builtins courants (len, str, int, float, bool, abs, min, max, sum, round, sorted). Plus les méthodes d'aide `has()`, `delete()`, `keys()`.

### Modifié

- **Modèle de visibilité basé sur l'abonnement** — `build_visibility_filter()` simplifié de 3 niveaux (propre → membre org → abonné) à 2 niveaux (propre → abonné). La migration de données crée automatiquement des lignes `ResourceSubscription` pour tous les membres org existants afin de préserver l'accès. Le nombre de connecteurs du tableau de bord et les statistiques KB incluent désormais les ressources abonnées.
- **Paramètres de portée et de catégorie de l'API Market** — `browse_market()` accepte les paramètres de requête `scope` (market ou org:`{id}`) et `category` (solutions ou components). `subscribe_resource()` valide désormais l'appartenance à l'org pour les abonnements org non-market.

- **Refactorisation de la liste blanche du cache d'outils** — Remplacement de la liste noire codée en dur dans `ToolCache` par une approche de liste blanche. Ajout de la propriété `cacheable` au protocole `Tool` (par défaut `False`) ; seuls les outils explicitement marqués `cacheable=True` sont mis en cache. 11 outils en lecture seule (web_search, kb_retrieve, calculator, etc.) marqués cacheable ; tous les outils avec effets secondaires (code exec, email, connecteurs) restent non mis en cache par défaut. Les nouveaux outils connecteur sont sûrs sans configuration supplémentaire.
- **Défaillance en cascade de l'exécuteur DAG** — Les étapes échouées ou expirées bloquent désormais en cascade leurs dépendants. Ajout du suivi `failed_ids` avec propagation transitive (chaînes A→B→C). Les étapes dépendantes sont automatiquement marquées comme échouées avec un message descriptif au lieu de s'exécuter avec un contexte empoisonné.
- **Correction du statut du vérificateur d'étape DAG** — Les échecs de nouvelle tentative de vérification d'étape marquent désormais correctement l'étape comme `"failed"` au lieu de la laisser comme `"completed"`. Les vérifications échouées se propagent aux étapes en aval via le nouveau mécanisme de défaillance en cascade.
- **Descriptions d'outils du planificateur DAG** — Le planificateur reçoit désormais `(name, description)` pour chaque outil au lieu de simplement les noms, permettant un mappage étape-vers-outil plus précis et réduisant les valeurs `tool_hint` halluccinées.
- **Historique complet du re-planificateur DAG** — Le contexte de re-planification accumule désormais les résultats de TOUS les tours précédents (pas seulement le dernier). Les tours plus anciens sont tronqués plus agressivement (200 caractères/étape) tandis que le tour le plus récent conserve 500 caractères/étape.
- **Qualité de synthèse de stream_answer** — Augmentation de la troncature d'observation d'outil de 2000 à 8000 caractères lors de la synthèse des réponses finales, préservant plus de données structurées (tableaux JSON, sortie de code).
- **14 constantes de moteur paramétrées** — Extraction des constantes numériques codées en dur vers des variables d'environnement dans l'agent ReAct, le planificateur DAG, le garde de contexte, l'espace de travail et les modules système.
