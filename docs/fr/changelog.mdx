## [Non publié]

### Ajouts

- **Délai d'expiration au niveau du flux de travail** — Nouveau paramètre `workflow_timeout_ms` dans le moteur qui annule tous les nœuds en attente/en cours lorsqu'il est dépassé, avec attente asynchrone bornée pour une détection de délai d'expiration réactive.
- **Réexécution avec les dernières entrées** — Le bouton "Exécuter à nouveau" pré-remplit le formulaire d'entrée avec les valeurs de l'exécution précédente.
- **Filtre de statut de l'historique d'exécution** — Menu déroulant pour filtrer les exécutions de flux de travail par Terminé/Échoué/Annulé/En cours.
- **Historique des versions du flux de travail** — Versioning automatique lors de l'enregistrement du schéma avec des points de terminaison API list/get/restore pour revenir à des versions antérieures.
- **Point de terminaison de validation** — `POST /validate` pour la validation structurelle du schéma sans exécution.
- **Mode d'exécution à blanc** — `POST /run` avec `dry_run=true` retourne le plan d'exécution en JSON sans exécuter.
- **13 tests de routage de branche conditionnelle** — Couverture complète pour le branchement, la fusion, l'imbrication et le routage du classificateur.
- **Panneau de statistiques d'exécution par nœud** — Nouveau panneau de statistiques (`node-stats-panel.tsx`) affichant les taux de succès par nœud, les durées moyennes et les compteurs d'échecs triés du pire au meilleur. Intégré à la barre d'outils du flux de travail. Nouvelle méthode client API `getNodeStats` et type `NodeStatEntry` pour la consommation frontend.
- **Nœuds de phase 3 : SubWorkflow et ENV** — Deux nouveaux types de nœuds avec exécuteurs backend, validation du parseur et 14 nouveaux tests (total : 304 → 306) :
  - **SubWorkflow** : Exécuteur stub pour l'orchestration de flux de travail imbriqués — sélectionnez un flux de travail cible et mappez les variables d'entrée/sortie.
  - **ENV** : Lit les variables d'environnement chiffrées à l'exécution avec sélection de clé et valeurs par défaut de secours.
- **Composants frontend SubWorkflow et ENV** — Intégration frontend complète : `sub-workflow-node.tsx` et `env-node.tsx` avec formulaires de configuration, entrées de palette, couleurs de minimap et entrées d'enregistrement de type de nœud dans tous les mappages de type.
- **Deux nouveaux modèles de flux de travail (14 au total)** — "Intégration API sécurisée" (ENV → HTTP avec en-têtes d'authentification) et "Orchestration de sous-flux de travail" (LLM → SubWorkflow → résumer).
- **Dialogue des raccourcis clavier** — Appuyez sur `?` pour afficher tous les raccourcis clavier disponibles dans une superposition de dialogue.
- **Capture d'aperçu d'entrée de nœud** — Les événements d'exécution incluent maintenant du JSON tronqué des sorties des prédécesseurs pour le débogage.
- **Point de terminaison d'analyse du flux de travail** — `GET /analytics` retourne le nombre total d'exécutions, la distribution des statuts, le taux de succès, les percentiles de durée p50/p95/p99, les tendances quotidiennes et les nœuds les plus défaillants.
- **Panneau du tableau de bord d'analyse** — Cartes de statistiques, barre de distribution des statuts, graphique quotidien en CSS pur, barres de percentiles et tableau des nœuds défaillants avec sélecteur de plage 7j/14j/30j.
- **Dialogue de galerie de modèles** — Navigation en grille visuelle des modèles de flux de travail avec filtres de catégorie.
- **Bords étiquetés** — Noms de branche conditionnelle et classificateur affichés sous forme de pastilles sur les bords.
- **Panneau de variables du flux de travail** — Définir, modifier et supprimer les variables au niveau du flux de travail avec type/défaut/description.

### Ajouté

- **Retry de nœud du moteur de flux de travail** — Retry par nœud avec backoff exponentiel. Les nœuds peuvent spécifier `retry_count` et `retry_delay_ms` dans leur configuration de données. Les nœuds défaillants sont relancés jusqu'à N fois avec un délai plafonné à 60s. Émet des événements SSE `node_retrying` avec le numéro de tentative et l'erreur précédente. Respecte les événements d'annulation pendant l'attente de retry.
- **Évaluation d'expression du magasin de variables de flux de travail** — Évaluation sécurisée d'expression Python via simpleeval. La méthode `evaluate_expression()` expose les variables du magasin (excluant env) avec les builtins courants (len, str, int, float, bool, abs, min, max, sum, round, sorted). Plus les méthodes d'aide `has()`, `delete()`, `keys()`.
- **Nœud de phase 2 du flux de travail : HumanIntervention** — Nœud de pause pour approbation (stub d'approbation automatique). Lit prompt_message (avec interpolation {{variable}}), assignee, timeout_hours. Nœud frontend à thème Sky-500 avec aperçu de prompt et badge @assignee. Panneau de configuration avec textarea de prompt, entrée assignee, champ timeout.
- **Modèles de flux de travail étendus à 12** — Quatre nouveaux modèles : Human Approval Pipeline (LLM → HumanIntervention → ConditionBranch → dual End), Multi-Step Data Processing (CodeExecution → ConditionBranch → Transform → ListOperation), Knowledge-Enhanced Agent (QuestionUnderstanding → KnowledgeRetrieval → LLM), Iterative Review Loop (Loop → LLM → VariableAssign).
- **Menu contextuel du volet de flux de travail** — Clic droit sur le canevas vide pour Paste (désactivé quand le presse-papiers est vide), Select All, Fit View, et actions Auto Layout. L'état d'union discriminée sépare les menus contextuels de nœud et de volet.
- **Menu contextuel du nœud de flux de travail** — Clic droit sur n'importe quel nœud pour accéder aux actions Configure, Copy, Duplicate, et Delete. L'action Delete est masquée pour les nœuds singleton Start/End. Le menu se ferme automatiquement au clic extérieur ou au défilement.
- **Modèles de phase 2 du flux de travail** — Deux nouveaux modèles intégrés (8 au total) : List Processing Pipeline (Start → ListOperation → Transform → End) et Enhanced Question Answering (Start → QuestionUnderstanding → LLM → End).
- **Validation de phase 2 du flux de travail** — Avertissements de validation de blueprint étendus pour ListOperation (input/expression manquants), Transform (input manquant/opérations vides), Iterator (variable de liste manquante), Loop (condition manquante), DocumentExtractor, QuestionUnderstanding (input manquant), ParameterExtractor (texte d'entrée/paramètres vides manquants). Correction : l'opération de tri ne nécessite plus incorrectement une expression.
- **Notes de nœud de flux de travail** — Fonctionnalité d'annotation par nœud. Cliquez sur « Note » dans le panneau de configuration de n'importe quel nœud pour ajouter une note en texte libre affichée en italique sous le nœud sur le canevas. Textarea auto-extensible avec stockage `data.note`.
- **Magnétisme à la grille du flux de travail** — Grille d'accrochage 16px correspondant au motif de fond pointillé pour un alignement précis des nœuds.
- **Validation de connexion du flux de travail** — Empêche les auto-connexions et les arêtes dupliquées en plus des restrictions Start/End existantes.
- **Dépôt sensible au zoom du flux de travail** — Le glisser-déposer de la palette de nœuds utilise `screenToFlowPosition()` pour un placement correct à n'importe quel niveau de zoom.
- **Données multi-lignes SSE du flux de travail** — L'analyseur SSE gère désormais les champs `data:` multi-lignes et ignore les lignes de commentaire keepalive.
- **Raccourcis clavier du flux de travail** — Copy (Cmd+C), paste (Cmd+V), duplicate (Cmd+D), select all (Cmd+A), Escape pour désélectionner. Copy/paste génère de nouveaux IDs et décale les positions ; les nœuds Start/End sont exclus de la copie pour maintenir la contrainte singleton.
- **Édition de description en ligne du flux de travail** — Cliquez sur la ligne de description dans la barre d'outils pour ouvrir un popover textarea pour éditer les descriptions de flux de travail sans naviguer.
- **Mode LLM ConditionBranch du flux de travail** — Évaluation de condition appropriée basée sur LLM remplaçant le stub naïf « match first ». Construit une invite de classification à partir des étiquettes de condition, appelle le LLM rapide, et utilise la correspondance insensible à la casse + floue.
- **Étiquettes d'arête réactives** — Les étiquettes d'arête pour les branches de condition/classificateur utilisent désormais `useNodesData()` pour les mises à jour en temps réel lorsque les conditions ou les classes sont éditées dans le panneau de configuration.
- **Panneau de statistiques du flux de travail** — Affichage de statistiques basé sur feuille montrant le total des exécutions, les comptes complétés/échoués/annulés, le taux de succès avec barre de progression visuelle, la durée moyenne, et l'heure de la dernière exécution avec formatage relatif. Déclenché à partir d'une icône BarChart3 dans la barre d'outils.
- **Nœuds de flux de travail de phase 2 : DocumentExtractor & QuestionUnderstanding** — Deux autres types de nœuds :
  - **DocumentExtractor** : Exécuteur stub pour le traitement de texte de document — supporte les types d'entrée text/base64 et 4 modes d'extraction (full_text, pages avec page_range, metadata avec comptes word/char/line/page, tables avec détection de table markdown). L'entrée URL est reportée pour une implémentation ultérieure.
  - **QuestionUnderstanding** : Prétraitement de question basé sur LLM avec 4 modes — rewrite (clarté), expand (contexte + sous-questions), classify (JSON intent/topic/confidence), decompose (tableau de sous-questions). Utilise le LLM rapide avec remplacement de prompt système personnalisé optionnel.
  - Frontend complet : composants de nœud, panneaux de configuration, entrées de palette (catégorie AI pour QU, catégorie Data pour DE), support du sélecteur de variables.
- **Nœuds de flux de travail de phase 2 : ListOperation & Transform** — Deux types de nœuds de traitement de données :
  - **ListOperation** : 8 opérations de liste (filter, map, sort, slice, flatten, unique, reverse, length) avec expressions simpleeval pour filter/map/sort. Supporte l'analyse automatique d'entrée de chaîne JSON.
  - **Transform** : Transformation de données basée sur pipeline avec 6 types d'opération (json_path, type_cast, format, regex_extract, string_op, math_op). Les opérations sont appliquées séquentiellement — la sortie d'une alimente la suivante.
  - Frontend complet : composants de nœud, panneau de configuration avec éditeurs spécifiques à l'opération, entrées de palette, support du sélecteur de variables.
- **Nœuds de flux de travail de phase 2 : Loop & ParameterExtractor** — Deux autres types de nœuds avancés :
  - **Loop** : Boucle while-condition avec évaluation d'expression simpleeval, nom de variable de boucle configurable, et limite de sécurité max_iterations. Frontend avec entrée de condition, nom de variable, et configuration de limite d'itération.
  - **ParameterExtractor** : Extraction de paramètres structurés basée sur LLM à partir du texte. Définissez les paramètres typés (name/type/description/required), fournissez la référence de texte d'entrée, et optionnellement ajoutez les instructions d'extraction. Retourne JSON analysé. Frontend avec éditeur de liste de paramètres et textarea de prompt d'extraction.
- **Nœuds de flux de travail de phase 2 : Iterator & VariableAggregator** — Deux nouveaux types de nœuds avancés :
  - **Iterator** : Itération sur les listes avec noms de variables item/index configurables et limite de sécurité max_iterations. Supporte les listes natives, l'analyse de chaîne JSON, et les références de variables de modèle.
  - **VariableAggregator** : Fusionner les sorties de plusieurs branches en amont via quatre modes — collecter dans une liste, concaténer des chaînes (séparateur personnalisé), fusionner des dicts, ou prendre la première valeur non vide.
  - Intégration frontend complète : composants de nœud, entrées de palette, formulaires de panneau de configuration, couleurs de minimap, support du sélecteur de variables. Exécuteurs backend enregistrés dans le moteur.
- **Modèles de flux de travail étendus** — Ajout de deux nouveaux modèles intégrés (6 au total) : Data Processing Pipeline (Start → CodeExecution → TemplateTransform → End) et Agent with Knowledge Retrieval (Start → KnowledgeRetrieval → Agent → End). Correction des champs `icon` manquants sur les nouveaux modèles. Ajout des clés i18n manquantes pour les catégories de modèle « intermediate » et « advanced ».
- **Dialogue des variables ENV du flux de travail** — Gestion de l'interface utilisateur des variables d'environnement chiffrées par flux de travail. Éditeur clé-valeur avec affichage masqué, détection de clé dupliquée, et validation en ligne. Connecté aux points de terminaison `GET/PUT /api/workflows/{id}/env`. Accessible via le menu déroulant de la barre d'outils.
- **Sauvegarde automatique du flux de travail** — Minuteur de sauvegarde automatique débounce de 5 secondes avec remplacement immédiat Cmd+S. L'indicateur d'état de sauvegarde affiche « Saving... », « Unsaved changes », ou « Saved Xs ago » avec actualisation automatique de 30s.
- **Recherche et filtrage de la liste de flux de travail** — Recherche côté client par nom/description, boutons de filtre d'état (All/Draft/Active), menu déroulant de tri (Recently updated/created/Name A-Z), et état vide pour aucun résultat.
- **MiniMap codé par couleur du flux de travail** — Les couleurs des nœuds MiniMap correspondent désormais à la couleur de catégorie de chaque type de nœud (vert=Start, rouge=End, bleu=LLM, orange=Condition, etc.).
- **Système de compétence d'agent** — Chargement de compétence à la demande pour les instructions d'agent : modèle `Skill` (name, description, content, script) avec CRUD complet, examen de publication, et cycle de vie de visibilité ; outil intégré `read_skill(name)` pour la divulgation progressive (~80% de réduction de token) ; stubs de compétence injectés dans le prompt système (~15 tokens/compétence) ; champ `compact_instructions` par agent pour la stratégie de compaction ContextGuard personnalisée.
- **Interface utilisateur des compétences** — Page de liste de compétences, éditeur pleine page (onglets content/script/settings), composant SkillCard, navigation de barre latérale, sélecteur de compétence d'éditeur d'agent avec textarea d'instructions compactes.
- **Flux de publication du flux de travail** — Points de terminaison publish/unpublish/resubmit pour les flux de travail, correspondant au modèle de publication d'agent existant. Nouveau drapeau `review_workflows` sur le modèle Organization (avec migration Alembic) pour activer la gestion de révision au niveau de l'organisation. Les actions de publication sont câblées dans l'interface utilisateur de workflow-card et de barre d'outils.
- **Système de blueprint de flux de travail** — Éditeur de flux de travail visuel pour concevoir et exécuter des blueprints d'automatisation multi-étapes :
  - **12 types de nœuds** : Start, End, LLM, ConditionBranch, QuestionClassifier, Agent, KnowledgeRetrieval, Connector, HTTPRequest, VariableAssign, TemplateTransform, CodeExecution — chacun avec un composant de nœud React Flow personnalisé, une icône spécifique au type, et un formulaire de panneau de configuration dédié.
  - **Moteur d'exécution** : `WorkflowEngine` avec tri topologique pour l'ordre d'exécution des nœuds, contrôle de concurrence basé sur `asyncio.Semaphore`, et branchement de condition/classificateur qui active/désactive les arêtes en aval en fonction de `active_handles`.
  - **Streaming SSE** : Progression d'exécution de flux de travail en temps réel via Server-Sent Events — émet les événements `run_started`, `node_started`, `node_completed`, `node_failed`, `node_skipped`, `run_completed`, et `run_failed`. Keepalive SSE pour les flux de travail de longue durée ; la détection de déconnexion du client déclenche l'annulation.
  - **Magasin de variables** : `VariableStore` thread-safe avec interpolation `{{node_id.output}}`, résolution de nom plat, espace de noms `env.*` pour les variables d'environnement chiffrées, `snapshot_safe()` (exclut les secrets), et `list_available_variables()` pour le sélecteur frontend.
  - **Étiquettes d'arête** : `WorkflowEdgeDef` porte `source_handle` et `target_handle` pour les branches de condition/classificateur, permettant les arêtes étiquetées qui indiquent visuellement quel chemin de branche chaque connexion représente.
  - **Combobox du sélecteur de variables** : Combobox consultable (popover basé sur `Command`) pour insérer les références `{{node_id.output}}` des nœuds en amont — groupées par type de nœud avec icônes, intégrées dans tous les champs prompt/template/body.
  - **Feuille d'historique d'exécution** : Feuille latérale affichant les exécutions de flux de travail passées avec l'état, la durée, les données d'entrée/sortie, et les résultats par nœud. Paginée via `GET /api/workflows/{id}/runs`.
  - **Duplication de flux de travail** : Point de terminaison `POST /api/workflows/{id}/duplicate` et action d'interface utilisateur pour cloner un flux de travail existant (copie blueprint, schema, description ; réinitialise l'état à draft).
  - **Validation de blueprint** : Point de terminaison `POST /api/workflows/validate` — valide un blueprint sans enregistrer (vérifie la présence des nœuds Start/End, les IDs dupliqués, les types de nœuds inconnus, les références d'arête, la détection de cycle). Retourne les comptes de nœud/arête en cas de succès ou les erreurs de validation spécifiques.
  - **Stratégies d'erreur** : Champ `error_strategy` par nœud avec trois modes — `STOP_WORKFLOW` (par défaut : arrêter tous les nœuds restants), `CONTINUE` (marquer comme échoué mais laisser les nœuds en aval continuer), `FAIL_BRANCH` (ignorer tous les nœuds accessibles uniquement via le nœud échoué).
  - **Timeout par nœud** : `timeout_ms` configurable par nœud (par défaut 30s) appliqué via `asyncio.wait_for` ; le timeout produit un résultat structuré `FAILED` avec un message d'erreur descriptif.
  - **Panneau de configuration du nœud** : Formulaires de configuration par type — les nœuds LLM ont prompt système, sélecteur de modèle, bascule de niveau de modèle (fast/main), curseur de température, max tokens ; ConditionBranch a des menus déroulants d'opérateur (==, !=, >, <, contains, etc.) avec mode expression/LLM ; HTTPRequest a sélecteur de méthode, URL, éditeur d'en-têtes (ajouter/supprimer des paires clé-valeur), body avec insertion de variable ; CodeExecution a sélecteur de langage (Python/JavaScript) avec des indices appropriés à la syntaxe ; tous les nœuds de sortie incluent `OutputVariableField` et `InsertVariableBar`.
  - **Add-Node-on-Edge** : Composant `AddNodeEdge` personnalisé avec bouton "+" au point médian — cliquer ouvre un popup de sélecteur de type de nœud qui insère un nouveau nœud en ligne, divisant l'arête existante en deux connexions.
  - **Auto-Layout (ELK.js)** : Utilitaire `getAutoLayoutedNodes()` utilisant l'algorithme en couches ELK.js avec `elk.direction: RIGHT` pour l'arrangement de nœud horizontal de gauche à droite. Espacement de nœud configurable (60px entre les nœuds, 120px entre les couches).
