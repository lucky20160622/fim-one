> Objectif : Construire un **Hub de Connecteurs alimenté par l'IA** — Autonome (assistant de portail), Copilote (intégré au système hôte), Hub (orchestration centrale inter-systèmes).
>
> Principes : **Agnostique des fournisseurs** (pas de verrouillage propriétaire), **abstraction minimale**, **orienté protocole**, **orienté connecteur** (l'intégration est la valeur fondamentale).

## Vision du Produit

FIM One est un **Hub de Connecteurs IA** qui propose trois modes progressifs :

```
Standalone   → Votre propre assistant IA (Portal)
Copilot      → IA intégrée dans un système hôte (iframe / widget / embed)
Hub          → Orchestration centrale inter-systèmes (Portal / API)
```

**Le mode Hub est le différenciateur clé.** Les clients entreprise disposent de systèmes hérités — ERP, CRM, OA, finance, HR — qui doivent communiquer entre eux via l'IA :

```mermaid
flowchart LR
  subgraph Sources["Systèmes Source"]
    ERP["ERP<br/>(SAP/Kingdee)"]
    CRM["CRM<br/>(Salesforce)"]
    OA["OA<br/>(Seeyon/Weaver)"]
    DB["Base de données personnalisée<br/>(PG/MySQL)"]
  end

  subgraph Hub["FIM One Hub"]
    A["Agent A: Audit Financier"]
    B["Agent B: Révision de Contrat"]
    C["Agent C: Assistance d'Approbation"]
    D["Agent D: Rapports de Données"]
  end

  subgraph Targets["Cibles de Notification"]
    T1["Lark / Slack"]
    T2["Email / WeCom"]
    T3["Teams / Webhook"]
    T4["Toute API"]
  end

  subgraph Delivery["Livraison"]
    Portal["Portal (UI)"]
    API["API (headless)"]
    Embed["iframe (embed)"]
  end

  ERP --> A --> T1
  CRM --> B --> T2
  OA --> C --> T3
  DB --> D --> T4
  Hub --> Delivery
```

**Stratégie GTM : Land and Expand**

| Étape | Mode | Ce qui se passe |
|------|------|-------------|
| Land | Copilot | Intégrer dans un système, prouver la valeur dans leur interface utilisateur |
| Expand | Copilot → Hub | Déployer sur plus de systèmes ; Hub les agrège |

Hooks définis par l'utilisateur : configuration YAML par agent (`hooks:` champ) déclarant des commandes shell ou des callables Python déclenchés sur les événements d'outil correspondants — même modèle que les hooks de Claude Code. Principe de conception clé : **les hooks sont pour la logique "doit toujours se produire" qui ne devrait jamais dépendre du LLM pour s'en souvenir**. Les instructions disent « enregistrer tous les appels » ; les hooks les enregistrent réellement. Les instructions disent « ne pas écrire en mode lecture seule » ; les hooks le bloquent réellement.

Ordre d'implémentation par complexité :
  1. **Fork du serveur MCP** — le plus simple ; copier la configuration (commande, args, modèle d'env). L'override des identifiants par utilisateur pour les serveurs MCP (`mcp_server_credentials`) fournit déjà une isolation d'identifiant atomique — étendre ce modèle
  2. **Fork de Skill** — copier le nom, le contenu/SOP, les scripts
  3. **Fork d'agent** — copier la configuration + remapper `connector_ids`, `kb_ids`, `sub_agent_ids`, `mcp_server_ids` aux copies bifurquées (nécessite que les ressources feuilles soient bifurquées en premier). Remarque : les Skills sont maintenant globaux et ne sont plus liés aux agents — le fork de Skill est indépendant
  4. **Fork de connecteur** — déjà livré (v0.8) ; vérifier qu'il gère l'élimination des identifiants lors du fork (l'utilisateur doit fournir ses propres identifiants)
  5. **Fork de workflow** — la duplication existe déjà ; vérifier qu'elle gère les références de nœud aux agents/connecteurs/KBs
  6. **Fork de KB** — le plus complexe ; **copie superficielle uniquement** (métadonnées + références de document, embeddings régénérés). La copie profonde des indices vectoriels est prohibitivement coûteuse et gaspilleuse puisque les utilisateurs de fork modifient généralement les documents. Référence l'approche npm : ne pas sur-concevoir, superficiel est suffisant
  **Stratégie de câblage du fork** : tri topologique sur le graphe de dépendance des ressources — bifurquer d'abord les nœuds feuilles (KB, Connecteur, MCP), construire une table de mappage `old_id → new_id`, puis bifurquer les nœuds parents (Agent, Skill) avec substitution d'ID. Réutiliser les modèles du résolveur de conflit d'importation de workflow (`compute_blueprint_diff()` mappage d'ID). Cette Phase 1 est indépendamment utile — les utilisateurs peuvent bifurquer des ressources individuelles du marché sans avoir besoin d'un package
