모든 구성은 `.env`를 통해 수행됩니다. `example.env`를 복사하고 값을 입력하세요:

```bash
cp example.env .env
```## 구성 수준

각 통합에는 중요도를 나타내는 구성 수준이 있습니다:

| 수준 | 의미 | 구성되지 않았을 때의 동작 |
|-------|---------|------------------------------|
| **Required** | 핵심 시스템 종속성 | 시스템이 오류 발생 — 채팅 및 주요 기능이 작동하지 않음 |
| **Recommended** | 중요한 기능 활성화 | 우아한 성능 저하 — 기능이 눈에 띄게 사용 불가능하지만 시스템은 실행됨 |
| **Optional** | 향상 기능 | 투명한 성능 저하 — 시스템이 정상 작동하며, 기능만 없음 |

> **참고**: 관리자가 구성한 모델(Admin → Models 페이지)은 LLM 환경 변수를 대체할 수 있습니다. 상태 확인은 두 소스를 모두 고려합니다.

---## Frontend (로컬 개발 전용)

프론트엔드는 **로컬 개발 전용** 별도의 env 파일을 가지고 있습니다: `frontend/.env.local`.

> **이 파일은 Docker에서 사용되지 않습니다.** Docker 컨테이너 내에서 Next.js는 `/api/*`를 Python 백엔드로 내부적으로 프록시합니다 (포트 8000은 컨테이너 내부용이므로), 프론트엔드 env 파일이 필요하지 않습니다.

로컬 개발의 경우 기본값이 그대로 작동합니다 — 백엔드가 기본이 아닌 포트에서 실행되지 않는 한 `frontend/.env.local`을 생성할 **필요가 없습니다**.

재정의가 필요한 경우 `frontend/.env.local`을 수동으로 생성하세요:

```bash
echo 'NEXT_PUBLIC_API_URL=http://localhost:9000' > frontend/.env.local
```

| 변수 | 기본값 | 설명 |
|---|---|---|
| `NEXT_PUBLIC_API_URL` | `http://localhost:8000` *(자동)* | **브라우저**가 직접 API 호출(OAuth 리다이렉트, 스트리밍)에 사용하는 백엔드 URL입니다. 설정되지 않은 경우 `window.location`에서 자동 감지됩니다 — 백엔드가 로컬에서 비표준 포트에서 실행되는 경우에만 재정의하세요. |

> **빌드 타임 참고**: `NEXT_PUBLIC_*` 변수는 `pnpm build` 타임에 JS 번들에 포함됩니다. 런타임에 변경하는 것(예: 루트 `.env`를 통해)은 효과가 없습니다 — 이것이 로컬 개발 전용으로 `frontend/.env.local`에 있는 이유입니다.## LLM (필수)

> **해결 순서**: 사용자 기본 설정 → Admin Models (DB) → ENV Fallback. Admin → Models에서 "General" 역할의 관리자 모델이 구성된 경우, 이 ENV 변수는 폴백으로만 제공됩니다. 상태 확인은 두 소스를 모두 고려합니다.### Extended Thinking (Reasoning)

When `LLM_REASONING_EFFORT` is set, FIM One enables the model's extended thinking capability so the internal chain-of-thought is surfaced in the UI "thinking" step. FIM One uses [LiteLLM](https://github.com/BerriAI/litellm) to translate the reasoning effort parameter into each provider's native format automatically.#### 지원되는 제공자

| 제공자 | `LLM_BASE_URL` | 작동 방식 | 추론 콘텐츠 반환? |
|---|---|---|---|
| **OpenAI** (o1 / o3 / o4-mini) | `https://api.openai.com/v1` | `reasoning_effort` 기본적으로 전송됨 | 예 |
| **Anthropic** (Claude 3.7+) | `https://api.anthropic.com/v1/` | LiteLLM이 `thinking` 매개변수를 사용하여 기본 Anthropic API를 통해 라우팅 | 예 |
| **Google Gemini** (2.5+) | `https://generativelanguage.googleapis.com/v1beta/openai/` | `reasoning_effort`를 호환성 엔드포인트에서 전송 | 예 |

LiteLLM은 `LLM_BASE_URL`에서 제공자를 자동으로 감지하고 올바른 API 형식으로 매핑합니다. 알 수 없는 URL은 OpenAI 호환으로 처리됩니다.#### 중요한 주의사항

<Warning>
**타사 프록시 / 커스텀 엔드포인트는 보장되지 않습니다.**
`LLM_BASE_URL`이 타사 API 프록시(예: OpenRouter, one-api, 커스텀 게이트웨이)를 가리키는 경우, LiteLLM은 URL을 기반으로 올바르게 라우팅하려고 시도합니다. 그러나 프록시가 비표준 형식을 예상하는 경우, 추론이 예상대로 작동하지 않을 수 있습니다. 프록시의 예상 매개변수 형식에 대해서는 프록시의 설명서를 참조하세요.
</Warning>#### 추론을 사용할 때의 온도 제약

일부 제공자는 추론이 활성화되었을 때 온도 제약을 부과합니다:

- **Anthropic**: 확장 사고가 활성화되면 `temperature=1`이 필요합니다. Anthropic을 확장 사고와 함께 사용하는 경우 `LLM_TEMPERATURE=1`을 **반드시** 설정해야 합니다 — Anthropic은 사고가 활성화되었을 때 다른 값을 거부합니다.
- **OpenAI GPT-5.x**: 항상 `temperature=1`만 지원합니다. LiteLLM의 `drop_params` 필터링이 이를 자동으로 처리합니다 — 지원되지 않는 온도 값은 자동으로 삭제됩니다. GPT-5.x의 경우 사용자 조치가 필요하지 않습니다.#### `LLM_REASONING_BUDGET_TOKENS` 작동 방식

이 변수는 **주로 Anthropic 경로에서 의미가 있습니다**. 설정되면 자동 계산된 예산을 재정의하고 LiteLLM을 통해 `thinking` 매개변수에서 `budget_tokens`로 전송됩니다. 설정되지 않으면 예산은 `LLM_MAX_OUTPUT_TOKENS` x 노력 비율에서 파생됩니다:

| `LLM_REASONING_EFFORT` | 예산 비율 | 예시 (max_tokens = 64000) |
|---|---|---|
| `low` | 20% | 12,800 tokens |
| `medium` | 50% | 32,000 tokens |
| `high` | 80% | 51,200 tokens |

최소 예산은 1,024 tokens입니다 (Anthropic의 하드 최소값).

OpenAI 및 Gemini의 경우, 제공자가 `reasoning_effort` 수준에 따라 내부적으로 토큰 할당을 처리합니다 — `LLM_REASONING_BUDGET_TOKENS`는 영향을 미치지 않습니다.

| Variable | Required | Default | Description |
|---|---|---|---|
| `REACT_MAX_ITERATIONS` | No | `20` | ReAct 요청당 최대 도구 호출 반복 횟수 |
| `MAX_CONCURRENCY` | No | `5` | DAG executor의 최대 병렬 단계 수 |
| `DAG_STEP_MAX_ITERATIONS` | No | `15` | 각 DAG 단계 내 최대 도구 호출 반복 횟수 |
| `DAG_MAX_REPLAN_ROUNDS` | No | `3` | 목표 달성 실패 시 최대 자율 재계획 시도 횟수 |
| `DAG_REPLAN_STOP_CONFIDENCE` | No | `0.8` | 목표 달성 불가능 신뢰도가 이 임계값을 초과할 때 재시도 중지 (`0.0` = 조기 중지 안 함, `1.0` = 모든 실패 시 중지) |
| `DAG_TOOL_CACHE` | No | `true` | 단일 DAG 실행 내에서 동일한 도구 호출 캐시. 키별 비동기 잠금으로 thundering herd 방지. 캐싱을 비활성화하려면 `false`로 설정 |
| `DAG_STEP_VERIFICATION` | No | `false` | 각 DAG 단계 완료 후 LLM 기반 품질 검사 활성화. 실패 시 검증 피드백과 함께 단계가 한 번 재시도됨. 지연 시간이 증가하지만 결과 정확도 향상 |
| `AUTO_ROUTING` | No | `true` | 자동 쿼리 분류 활성화. 빠른 LLM이 각 쿼리를 분석하고 ReAct(단일 단계) 또는 DAG(다중 단계) 실행 모드로 라우팅. `/api/auto` 엔드포인트가 이를 사용. 명시적 모드 선택을 요구하려면 `false`로 설정 |
