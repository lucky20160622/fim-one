## 모든 회사가 직면한 문제

50명 규모의 스타트업부터 포춘 500대 기업까지 — 모든 회사 뒤에는 수년에 걸쳐 축적된 복잡한 시스템들이 있습니다:

```
ERP (SAP, Kingdee, Yonyou)        — finance, inventory, supply chain
CRM (Salesforce, HubSpot)         — customers, deals, pipelines
OA  (Seeyon, Weaver, Feishu/Lark) — approvals, documents, workflows
HR  (Workday, BambooHR)           — payroll, org charts, leave
Custom databases                   — orders, assets, tickets
Legacy systems                     — "nobody dares touch it"
```

각 시스템은 **정보 사일로**입니다. 데이터는 한 곳에, 워크플로우는 다른 곳에, 승인은 또 다른 곳에 존재합니다. 사람들은 시스템 간에 복사-붙여넣기를 하고, 스크린샷을 이메일로 보내고, 자동으로 흘러야 할 데이터를 수동으로 조정하는 데 수시간을 소비합니다.

**모든 시스템 공급업체가 이제 자신의 AI를 제공합니다.** Feishu에는 AI가 있습니다. Salesforce에는 Einstein이 있습니다. SAP에는 Joule이 있습니다. 하지만 각각은 **자신의 벽 안에서만** 똑똑합니다. Feishu AI는 ERP 잔액을 확인할 수 없습니다. Salesforce Einstein은 OA 승인을 트리거할 수 없습니다. 이들은 설계상 도메인에 제한되어 있습니다 — 경쟁사와의 연결이 비즈니스 이익에 반하기 때문입니다.

## FIM One이 하는 일

FIM One은 모든 시스템을 연결하는 **외부, 제3자, 수평적 플랫폼**입니다.

<Tip>
**클라이언트는 코드를 변경할 필요가 없습니다.** FIM One은 기존 시스템에 능동적으로 연결되어 데이터베이스를 읽고, API를 호출하고, 알림을 전송합니다. 제공하는 자격 증명과 네트워크 액세스만 사용합니다.
</Tip>

제공하는 것:
- 데이터베이스 자격 증명 (읽기 전용 계정 권장)
- API 엔드포인트 및 인증 키
- 네트워크 화이트리스트 액세스

FIM One이 제공하는 것:
- **모든** 시스템을 이해하고 조율하는 AI
- 모든 작업이 감사되고, 모든 쓰기 작업이 인간의 확인으로 제어됨
- 데이터는 시스템에 유지됨 — FIM One은 인텔리전스 계층을 제공하며, 데이터 레이크가 아님

## 두 가지 모드, 하나의 플랫폼

| 모드 | 설명 | 시작하기 |
|------|-----------|------------|
| **Copilot** | 하나의 시스템에 내장된 AI — 사용자의 기존 UI에서 함께 작동 | 빠르게 가치 증명: ERP에 "Finance Copilot" 임베드 |
| **Hub** | 모든 시스템을 연결하는 중앙 AI 포털 — 크로스시스템 오케스트레이션 | 가치 확대: ERP + CRM + OA + Lark을 한 곳에서 연결 |

**Land and Expand**: 하나의 시스템에서 Copilot으로 시작하세요. 사용자는 새로운 도구를 배울 필요 없이 즉각적인 가치를 경험합니다. ROI가 증명되면 Hub를 설정하세요 — 모든 시스템이 AI를 만나는 중앙 포털입니다. Copilot은 계속 실행되고, Hub는 크로스시스템 인텔리전스를 추가합니다.

> Copilot은 하나의 시스템 내에서 가치를 증명합니다. Hub는 모든 시스템에 걸쳐 가치를 발휘합니다.

## 왜 그냥...이 아닐까요?

### ...MuleSoft 또는 기존 iPaaS를 사용할까요?

MuleSoft는 시스템을 시스템에 연결합니다. FIM One은 시스템을 **AI**에 연결합니다. 기존 iPaaS는 수개월의 커스텀 매핑, 전담 통합 팀, 그리고 6자리 계약이 필요합니다. FIM One은 당신의 데이터를 의미론적으로 이해합니다 — 자연어로 원하는 것을 설명하면, AI가 어떤 시스템을 쿼리할지, 그리고 결과를 어떻게 결합할지 파악합니다.

### ...ETL 파이프라인을 직접 작성하시나요?

ETL은 소스 시스템이 스키마를 업그레이드할 때마다 중단됩니다. FIM One은 라이브로 연결되므로 데이터 복사, 동기화 작업, 파이프라인 유지 관리가 필요 없습니다. 그리고 양방향으로 작동합니다: 한 시스템에서 읽고, 다른 시스템에 쓰며, 모든 쓰기 작업에 대해 사용자 확인을 받습니다.

### ...Dify, n8n 또는 워크플로우 빌더를 사용할까요?

워크플로우 빌더는 새로운 캔버스에서 비즈니스 로직을 **복제**하도록 요구합니다. 하지만 당신의 워크플로우는 이미 존재합니다 — OA, ERP, 승인 시스템에서 말입니다. 또 다른 순서도 편집기를 원하지 않습니다. 이미 실행 중인 시스템에 연결되는 AI를 원합니다.

FIM One은 워크플로우 로직을 복제하지 않습니다. 커넥터의 관점에서 모든 복잡한 작업 — 이체, 거부, 에스컬레이션, 공동 서명 — 은 매개변수가 있는 단일 API 호출로 축소됩니다. 대상 시스템이 상태 머신을 관리합니다. FIM One은 단지 API를 호출할 뿐입니다.

### ...AI 에이전트 플랫폼(Manus, AutoGPT) 사용?

자율 에이전트는 강력하지만 통제되지 않습니다. 실패한 API를 재시도하면서 토큰을 소모하고, 목표에서 벗어나며, 커스텀 통합 없이는 독점 시스템에 연결할 수 없습니다. FIM One은 **제한된 자율성**을 제공합니다 — 가드레일이 있는 동적 계획, 실패한 커넥터에 대한 서킷 브레이커, 쓰기 작업에 대한 인간 확인 게이트.

### ...각 벤더의 기본 제공 AI에 맡기면 어떨까요?

Feishu AI는 Feishu 내에서만 작동합니다. Salesforce Einstein은 Salesforce 내에서만 작동합니다. 둘 다 경계를 넘을 수 없습니다. **어떤 벤더도 진정한 크로스 시스템 통합을 구축하지 않을 것입니다** — 이는 그들의 종속성 유지 인센티브에 위배됩니다. FIM One은 그러한 이해 충돌이 없는 유일한 플레이어입니다.

## FIM One 연결 방식

FIM One은 전송과 거버넌스를 분리하는 3계층 아키텍처를 사용합니다:

```
FIM One Platform (AI 오케스트레이션, 멀티테넌트, UI)
    ↓
Connector Governance Layer (감사, 인증, 확인 게이트, 서킷 브레이커)
    ↓
Connector (API / Database / MCP protocol)
    ↓
Your Systems (ERP, CRM, OA, databases, custom APIs)
```

**거버넌스 레이어**가 엔터프라이즈 수준의 준비를 가능하게 합니다:

| 기능 | 역할 |
|-----------|-------------|
| **감사 로깅** | 모든 도구 호출 기록 — 누가, 무엇을, 언제, 결과 |
| **사용자 확인** | 쓰기 작업은 실행 전 명시적 사용자 승인 필요 |
| **서킷 브레이커** | 시스템이 다운되면 FIM One은 즉시 재시도를 중단하고 토큰 소비 방지 |
| **자격증명 암호화** | AES-GCM 암호화(저장 시), 사용자별 자격증명 격리 |
| **읽기 전용 강제** | 기본값은 읽기 전용이며, 쓰기 작업은 명시적으로 제어됨 |

에이전트는 SAP, PostgreSQL 데이터베이스, 또는 사용자 정의 REST API와 통신하는지 알거나 신경 쓸 필요가 없습니다. 커넥터는 표준화된 브릿지이므로 새로운 시스템을 추가하려면 커넥터 구성만 필요하고 코드 변경은 불필요합니다.

## 커넥터를 구축하는 세 가지 방법

새로운 시스템을 연결하기 위해 Python을 작성할 필요가 없습니다:

| 방법 | 시간 | 필요한 기술 |
|--------|------|---------------|
| **OpenAPI 스펙 가져오기** | 분 단위 | YAML/JSON 스펙 또는 URL 업로드 — 커넥터 자동 생성 |
| **AI 채팅 빌더** | 분 단위 | 자연어로 API 설명 — AI가 커넥터 구성 생성 |
| **MCP 에코시스템** | 분 단위 | 모든 MCP 서버 직접 연결 — 오픈소스 MCP 커뮤니티 기본 지원 |

엔터프라이즈 데이터베이스의 경우, FIM One은 PostgreSQL, MySQL, Oracle, SQL Server 및 중국 레거시 데이터베이스(DM, KingbaseES, GBase, Highgo)에 대한 직접 SQL 액세스를 지원하며, 스키마 내부 검사, AI 기반 주석 처리 및 읽기 전용 쿼리 실행을 제공합니다.

## 지금이 가능한 이유

FIM One을 오늘날 가능하게 만드는 두 가지 변화가 있습니다:

**1. AI 모델이 드디어 오케스트레이션을 할 수 있을 정도로 똑똑해졌습니다.** GPT-4, Claude, Gemini는 기본 도구 호출, 긴 컨텍스트 윈도우(200K+), 동적 크로스 시스템 오케스트레이션을 안정적으로 만드는 추론 기능을 갖추고 있습니다. 3년 전에는 이것이 연구 문제였습니다. 오늘날에는 엔지니어링 문제입니다.

**2. 연결 계층은 여전히 아무도 담당하지 않습니다.** 모델은 계속 똑똑해지지만, 데이터베이스 자격증명을 기본적으로 암호화하거나, OAuth 토큰을 관리하거나, RBAC 정책을 적용하거나, 도구 호출을 감사할 수 없습니다. 그것은 인프라 작업이며, FIM One이 하는 작업입니다.

> 순수 LLM 오케스트레이션에서 엔터프라이즈 시스템 통합으로 갈수록 경쟁 우위는 더욱 견고해집니다.

AI 모델은 지능을 소유합니다. FIM One은 연결을 소유합니다. 더 똑똑한 모델은 FIM One을 **더 좋게** 만들지, 불필요하게 만들지 않습니다. 왜냐하면 더 나은 추론은 동일한 커넥터를 통한 더 나은 크로스 시스템 오케스트레이션을 의미하기 때문입니다.
