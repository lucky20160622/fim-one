> 목표: **AI 기반 커넥터 허브** 구축 — 독립형(포털 어시스턴트), 코파일럿(호스트 시스템에 임베드됨), 허브(중앙 크로스 시스템 오케스트레이션).
>
> 원칙: **공급자 중립적**(벤더 종속성 없음), **최소 추상화**, **프로토콜 우선**, **커넥터 우선**(통합이 핵심 가치).

## 제품 비전

FIM One은 **AI 커넥터 허브**로서 세 가지 점진적 모드를 제공합니다:

```
Standalone   → 자신의 AI 어시스턴트 (Portal)
Copilot      → 호스트 시스템에 내장된 AI (iframe / widget / embed)
Hub          → 중앙 집중식 크로스 시스템 오케스트레이션 (Portal / API)
```

**Hub 모드가 핵심 차별화 요소입니다.** 엔터프라이즈 클라이언트는 ERP, CRM, OA, 재무, HR 등의 레거시 시스템을 보유하고 있으며, 이들이 AI를 통해 서로 통신해야 합니다:

```mermaid
flowchart LR
  subgraph Sources["소스 시스템"]
    ERP["ERP<br/>(SAP/Kingdee)"]
    CRM["CRM<br/>(Salesforce)"]
    OA["OA<br/>(Seeyon/Weaver)"]
    DB["Custom DB<br/>(PG/MySQL)"]
  end

  subgraph Hub["FIM One Hub"]
    A["에이전트 A: 재무 감사"]
    B["에이전트 B: 계약 검토"]
    C["에이전트 C: 승인 지원"]
    D["에이전트 D: 데이터 리포팅"]
  end

  subgraph Targets["알림 대상"]
    T1["Lark / Slack"]
    T2["Email / WeCom"]
    T3["Teams / Webhook"]
    T4["Any API"]
  end

  subgraph Delivery["전달"]
    Portal["Portal (UI)"]
    API["API (headless)"]
    Embed["iframe (embed)"]
  end

  ERP --> A --> T1
  CRM --> B --> T2
  OA --> C --> T3
  DB --> D --> T4
  Hub --> Delivery
```

**GTM 경로: Land and Expand**

| 단계 | 모드 | 진행 상황 |
|------|------|-------------|
| Land | Copilot | 한 시스템에 내장하여 UI 내에서 가치 입증 |
| Expand | Copilot → Hub | 더 많은 시스템으로 확대; Hub가 이들을 통합 |

## 배포된 버전

### v0.1 (2026-02-22) — MVP: ReAct + DAG Planner
- ReActAgent with tools (calculator, python_exec, web_search)
- DAG Planner (LLM generates dependency graphs)
- Portal UI with streaming + KaTeX

### v0.2 (2026-02-24) — 다중 모델 + 메모리
- 재시도 / 속도 제한 / 사용량 추적
- 네이티브 함수 호출 (JSON 전용 파싱 없음)
- 다중 모델 지원 (빠른 + 메인 LLM)
- 메모리: WindowMemory, SummaryMemory
- SSE 스트리밍을 포함한 FastAPI 백엔드

### v0.3 (2026-02-25) — Web Tools + MCP
- Web tools (web_search, web_fetch) via Jina/Tavily/Brave
- File operations tool
- MCP client (standard tool integration)

- [ ] **리소스 포크 (패키지 Phase 1 — v1.0 패키지 시스템의 전제 조건)**: 리소스별 clone/fork 엔드포인트를 패키지 포크의 원자 빌딩 블록으로 사용합니다. 각 `POST /api/{type}/{id}/fork`는 원본과 분리된 사용자 소유의 리소스 구성 심층 복사본을 생성합니다 (업데이트 링크 없음). **테스트 매트릭스**: 각 리소스는 마켓 구독 (설치) 및 조직 수준 게시 (두 가지 별개의 코드 경로)에 대해 테스트되어야 합니다. 복잡도별 구현 순서:
  1. **MCP 서버 포크** — 가장 간단함; 구성 복사 (명령, 인수, 환경 템플릿). MCP 자격 증명 사용자별 재정의 (`mcp_server_credentials`)는 이미 원자 자격 증명 격리를 제공합니다 — 이 패턴을 확장합니다
  2. **스킬 포크** — 이름, 콘텐츠/SOP, 스크립트 복사
  3. **에이전트 포크** — 구성 복사 + `connector_ids`, `kb_ids`, `sub_agent_ids`, `mcp_server_ids`를 포크된 복사본으로 다시 매핑 (리프 리소스가 먼저 포크되어야 함). 참고: 스킬은 전역이며 더 이상 에이전트에 바인딩되지 않습니다 — 스킬 포크는 독립적입니다
  4. **커넥터 포크** — 이미 배포됨 (v0.8); 포크 시 자격 증명 제거를 처리하는지 확인 (사용자는 자신의 자격 증명을 제공해야 함)
  5. **워크플로우 포크** — 중복이 이미 존재; 에이전트/커넥터/KB에 대한 노드 참조를 처리하는지 확인
  6. **KB 포크** — 가장 복잡함; **얕은 복사만** (메타데이터 + 문서 참조, 임베딩 재생성). 벡터 인덱스를 심층 복사하는 것은 금지되어 있으며 포크 사용자가 일반적으로 문서를 수정하므로 낭비입니다. npm의 접근 방식을 참조: 과도하게 설계하지 마세요, 얕은 복사로 충분합니다
  **포크 배선 전략**: 리소스 종속성 그래프에 대한 위상 정렬 — 리프 노드 먼저 포크 (KB, 커넥터, MCP), `old_id → new_id` 매핑 테이블 구축, ID 대체를 통해 부모 노드 포크 (에이전트, 스킬). 워크플로우 가져오기 충돌 해결기 (`compute_blueprint_diff()` ID 매핑)의 패턴을 재사용합니다. 이 Phase 1은 독립적으로 유용합니다 — 사용자는 패키지가 필요 없이 마켓에서 개별 리소스를 포크할 수 있습니다

**영향**: FIM One을 자신감 있게 규모에 맞게 실행합니다. 네 가지 기둥: **추적 계층** (무엇이 일어났는지 확인), **훅 시스템** (발생해야 하는 것 강제), **에이전트 워크스페이스** (지속적 파일 관리 + 핸드오프), **IM 채널** (에이전트는 사용자가 작업하는 곳에 있음). 사전 구축된 솔루션 템플릿은 콜드 스타트를 제거합니다; 대시보드 개선은 운영 상태를 표시합니다. "에이전트가 따를 수 있는 지시"와 "시스템이 강제하는 보장" 사이의 간격이 닫혀 있습니다 — 데모와 프로덕션 엔터프라이즈 도구 간의 차이입니다.

| 메모리 라이프사이클 (TTL, 할당량) | 크로스 세션 메모리에 따라 달라짐; 함께 연기 |
| 활성 컨텍스트 압축 도구 (에이전트 트리거) | ContextGuard(v0.5)로 명시적 동결. 200K+ 컨텍스트 윈도우는 가치 감소. 컨텍스트 비용이 주요 엔터프라이즈 불만이 되지 않는 한 재검토하지 않음 |
| 브라우저 자동화 / 컴퓨터 사용 | 높은 유지보수 비용 (DOM 변경, 봇 방지, 샌드박싱). 업계가 컴퓨터 사용 모드(Anthropic, OpenAI Operator, Google Mariner)와 MCP 브라우저 도구(Puppeteer/Playwright MCP)로 수렴 중. MCP 통합을 통해 소비, 자체 구축 금지. 안정적인 컴퓨터 사용 MCP 표준이 나타날 때 재평가 |
